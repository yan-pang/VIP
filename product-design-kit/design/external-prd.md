# 领域长期对外交付 PRD 生成规格

> 默认输出到 `project/domains/<domain>/delivery/prd.md`。
>
> 领域长期 PRD 是 `design.md` 的**业务化对外投影**，持续反映当前有效设计，但不是人工维护的事实源。后续需求始终先更新 `design.md`，再按本规范重新生成 PRD。

## 一、固定生成链路

```text
design.md（完整业务、页面、交互与设计事实源）
        ↓ 去工程化、业务化转译
领域 delivery/prd.md（长期当前态，禁止直接编辑）
        ↓ 按版本冻结并增加变更摘要
project/delivery/v1.x/（版本交付快照）
```

- `design.md`：当前有效业务和产品设计的唯一事实源。
- `decisions.md`：历史取舍、覆盖关系和编号恢复依据，不覆盖当前设计。
- 领域长期 PRD：供业务、研发和测试共同评审的当前态需求文档。
- 版本快照：在长期 PRD 基础上增加本版范围、变更摘要、排期和评审信息。
- `project/research/`：提供背景、调研和外部依据；不直接成为当前产品规则。
- `project/tech/`：维护研发方案；只有用户可感知的约束才以业务语言进入 PRD。

当前项目未提供自动生成程序时，由执行者严格按本规范完成转译；不得因此把长期 PRD 改为人工事实源。

## 二、生成产物顶部说明

每份长期 PRD 必须以以下口径开头：

```markdown
> **生成产物，请勿直接编辑。**
>
> 当前业务事实源为 [`../design.md`](../design.md)。本文由该设计文档去除工程实现、原型状态与视觉细节后，以业务语言生成，面向业务、研发与测试持续提供最新有效需求。历史编号与变更原因参考 [`../decisions.md`](../decisions.md)，但历史记录不覆盖当前设计。
```

文档说明至少包括：

| 项目 | 要求 |
| --- | --- |
| 领域 | 当前领域中文名称 |
| 维护状态 | Active，始终反映当前有效设计 |
| 面向读者 | 业务、产品、研发、测试 |
| 需求编号段 | 当前领域已分配的稳定编号段 |
| 当前事实源 | 同领域 `design.md` |
| 历史追溯 | 同领域 `decisions.md` |
| 视觉规范 | `project/ui-brand.md` |

## 三、固定章节结构

长期 PRD 按以下顺序生成：

1. 文档说明及来源
2. 业务背景与核心目标
3. 角色与权限差异
4. 领域范围及跨领域边界
5. 业务对象与数据关系
6. 核心业务流程与状态机
7. 需求总览
8. 按稳定 P 编号组织的详细需求
9. 页面元素、状态与提示文案规格
10. 全局交互与非功能性要求
11. 迁移、暂缓与废弃需求索引

> 领域较简单时可裁剪第 9 章的元素范围,但只要领域含交互页面,第 9 章不得省略;第 6 章在含状态机的领域必须给出状态、转换和"不发生跃迁"的路径。

### 3.1 业务对象与数据关系

- 只描述业务对象、数量关系、归属关系和关键约束。
- 可以使用 Mermaid 关系图。
- 不出现表名、主外键、数据类型、存储方式或工程字段名。

### 3.2 核心业务流程与状态机

- 节点使用用户动作、业务判断和业务结果。
- 只保留研发和测试理解功能所需的主流程、异常分支和恢复路径。
- 页面实现流程、通信过程和内部任务调度不进入流程图。
- 领域含状态机时,除状态图外必须提供**状态转换矩阵**:原状态、触发事件、目标状态、负责人 / 归属变化、输入或操作资格、失败结果、恢复方式;并显式列出"**不发生跃迁**"的路径(如查看、发送失败、待发送、关系异常、账号离线)。
- 有独立结果状态的对象(如消息发送结果、导入结果)单独给出结果状态机。

### 3.3 长期范围

- 长期 PRD 不使用“本期包含”表达当前版本范围。
- “本领域负责”描述长期领域职责。
- “本领域不负责”描述长期归属边界。
- 临时版本范围和延期项由版本快照维护。

### 3.4 页面元素、状态与提示文案规格（第 9 章）

第 9 章用统一模板覆盖**全部功能元素**,不记录纯装饰性视觉细节。每个功能元素一行:

| 页面区域 | 元素 | 适用角色 | 显示条件 | 隐藏条件 | 启用条件 | 禁用条件 | 用户动作 | 即时反馈 | 成功结果 | 失败结果 | 最终文案 | 关联 P/R |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

- 至少覆盖:顶栏入口 / 告警 / 搜索 / 个人菜单、各类筛选器、列表分组与卡片、标题栏及全部操作按钮、内容项(如消息气泡)及其状态与操作、输入 / 附件 / 提交、各类弹窗、失败 / 详情面板、搜索面板,以及加载 / 空态 / 错误 / 断线 / 权限变化反馈。
- 跨领域挂载区只描述本领域拥有的展开 / 折叠 / 联动行为。
- 本章还必须包含**关键元素判定矩阵**(按元素维度拆解显示 / 启用 / 禁用 / 结果条件)与**最终提示文案字典**(见 3.6)。
- 不得使用“禁用并提示原因”“展示友好提示”“按统一规则处理”等不可验收表述。

### 3.5 多条件提示优先级

多个限制同时命中时,只展示**一个主原因**,并允许查看其他原因。PRD 必须给出该领域统一的优先级序列,同一元素不得因条件判断顺序不同而展示不同主提示。默认优先级(领域可细化,不可矛盾):

1. 权限或数据可见范围失效:隐藏入口或关闭当前内容。
2. 好友 / 对象关系异常。
3. 账号停用、封禁或离线。
4. IP 异常、频率上限等全局硬限制。
5. 业务状态或负责人不满足操作条件。
6. 网络断开或实时同步异常。
7. 云电脑、机器人等临时执行依赖不可用。
8. 当前内容自身的格式、大小或违禁词问题。

### 3.6 最终提示文案字典

- 提供可直接实现和验收的最终文案,动态内容使用明确占位符(如 `{客服姓名}`、`{关键词}`)。
- 每个占位符必须说明取值与**缺失时的降级文案**。
- 不向用户暴露内部错误码或原始技术错误。
- 至少覆盖:空态 / 未选择、各业务状态提示、账号与风控限制、关系异常与恢复、网络与同步、发送与待发 / 失败 / 违禁词、指派 / 冲突 / 权限变化、搜索、失败详情 / 录屏、加载 / 重试 / 确认 / 取消等文案族。

## 四、稳定编号与生命周期

### 4.1 P 编号

- 固定格式为 `P-XXX`。
- 首次生成前必须检查 Git 历史、旧 PRD、测试用例和决策记录。
- 已使用编号不得重新分配、合并复用或因标题变化而重排。
- 当前设计将多个历史需求组合到同一页面时，仍保留原有 P 条目。
- 新增需求使用该领域下一个从未使用的编号。

### 4.2 R 编号

- 规则编号固定为 `R-<P编号后三位>-NN`，例如 `R-108-03`。
- 当前语义仍有效时保留原编号并业务化重写。
- 规则迁移或失效时保留编号并说明去向，不得将该编号用于新规则。
- 新规则从该 P 条目下一个未使用的序号继续分配。

### 4.3 状态

| 状态 | 使用场景 |
| --- | --- |
| 有效，新增@v1.x | 首次引入且当前有效 |
| 有效，新增@v1.x；修订@v1.y | 当前有效，后续发生过语义变化 |
| 迁移@v1.x → `<domain> P-XXX` | 能力改由其他领域拥有 |
| 暂缓@v1.x | 能力保留编号,当前版本不提供入口 / 操作,后续版本恢复;可保留只读展示历史数据 |
| 废弃@v1.x | 当前不再提供且没有一一对应迁移项 |

迁移、暂缓和废弃条目保留原二级标题，并同时进入文末索引;暂缓条目只说明"当前不提供 + 保留编号 + 恢复条件",不描述当前可用操作,也不写实现差异或开发任务。

## 五、详细需求条目模板

每个条目以二级标题作为稳定锚点：

```markdown
## P-101 需求标题

**状态**：有效，新增@v1.0
**模块**：模块名称
**优先级**：P0
**来源**：路线图、调研或领域追加

### 业务目的与适用角色

### 前置条件与主流程

### 核心业务规则

| 规则编号 | 触发条件 | 预期行为 | 备注 |
| --- | --- | --- | --- |
| R-101-01 | 业务触发条件 | 用户可感知或可验证的结果 | 限制或去向 |

### 场景举例

### 页面与交互

### 边界与异常

### 跨领域依赖

### 验收标准
```

简单条目可合并小节,但每个**有效** P 条目必须完整覆盖以下内容(缺一即视为未达标):

- 状态、模块、优先级和来源。
- 业务目的、适用角色和前置条件。
- 主流程、备选流程和恢复流程。
- 至少一条稳定 `R` 编号的业务规则。
- 用户动作、即时反馈、处理中、成功结果和失败结果。
- 相关页面元素的显示、隐藏、启用和禁用条件(可指向第 9 章对应矩阵)。
- 状态判定及状态变化。
- 精确的提示文案(可指向第 9 章文案字典)。
- 跨领域依赖及其不可用时的影响。
- 可直接转成测试用例的验收矩阵(明确输入 / 条件 → 期望结果)。

以下表述**禁止**出现在有效条目中:"适当提示""友好提示""按统一规则处理""禁用并提示原因"等不可验收描述;凡涉及数值、时限、状态和文案,必须给出可验收的确定值。

## 六、去工程化转译规则

| Design 内容 | PRD 表达 |
| --- | --- |
| 用户动作、业务规则、状态、权限、异常 | 保留 |
| 文件大小、数量、期限、时效、结果上限 | 保留为可验收规格 |
| 数据对象 | 只保留业务名称和关系 |
| 接口、请求体、工程字段、内部错误码 | 改写为业务动作和结果 |
| 路由及查询参数 | 改写为页面跳转并定位业务对象 |
| 幂等机制 | 改写为重复操作不得造成重复业务结果 |
| 并发版本控制 | 改写为不得覆盖他人最新修改，冲突时提示刷新 |
| 长连接、缓存、本地存储 | 改写为自动同步、刷新后保持或不保持 |
| 框架、组件、事件处理器、原型数据 | 删除 |
| 像素、样式类、色值、组件库 | 留在 Design 或视觉规范 |
| 外部业务能力 | 保留依赖及业务影响，不描述协议和开发方式 |

### 6.1 页面与交互表达

使用“动作—反馈—结果”表达：

| 用户动作 | 即时反馈 | 成功结果 | 失败结果 |
| --- | --- | --- | --- |
| 用户提交操作 | 显示处理中并禁止重复提交 | 展示最新业务状态 | 保留原状态并说明原因 |

允许使用“列表”“详情面板”“确认弹窗”等通用产品语言；不使用工程组件名。

### 6.2 工程规则的业务化示例

设计原文：

> 历史加载后重新设置滚动位置，并用内部锁避免重复触发。

PRD：

> 用户向上浏览时逐轮加载更早历史；加载后保持当前阅读位置，加载过程中不得重复触发。

设计原文：

> 指派提交使用版本校验。

PRD：

> 多人同时指派时只能有一个结果生效；失败方看到最新负责人并收到冲突提示。

## 七、生成与验收检查

每次生成长期 PRD 时：

1. 读取完整 `design.md`，以当前稳定设计为准。
2. 读取现有长期 PRD、Git 历史和 `decisions.md`，仅用于恢复编号与变更轨迹。
3. 建立“有效设计能力 → P 条目 → R 规则”覆盖关系。
4. 对工程描述执行本规范的业务化转译。
5. 保留迁移和废弃编号。
6. 更新测试策略与用例的 P/R 引用。
7. 检查项目入口和本地链接。

质量门槛：

- 每项有效设计能力至少归属一个 P 条目。
- P/R 编号无重复、复用、丢失或语义错配。
- 每个有效 P 条目覆盖正常、权限、异常或恢复场景。
- 每个功能元素都有显示 / 启用 / 禁用 / 反馈 / 文案规格(第 9 章)。
- 每个禁用状态都有唯一主原因和确定的提示优先级。
- 每个异步操作都有处理中、成功、失败和恢复结果。
- 每个动态文案的变量及缺失降级均有定义。
- 不再出现"适当提示""友好提示""按统一规则"等模糊描述;数值、状态条件和提示文案可直接转为测试用例。
- 不出现接口路径、请求方法、数据库、数据类型、框架、组件、状态容器或原型数据说明。
- 产品可感知的限制、状态、时效和错误处理不得因去工程化而丢失。
- `git diff --check`、本地链接、Mermaid 语法和 Markdown 结构检查通过。
