# 领域长期 PRD 模板

> 默认保存为 `project/domains/<domain>/delivery/prd.md`。
> 这是领域的**长期 PRD**(需求条目的唯一事实源),持续演进、始终与最新版本相符,不随版本冻结。
> 对外交付物 `release-prd.md` 由 `/deliver` 按 `product-design-kit/design/release-prd-spec.md` **组装生成**(完整快照 + 本版变更摘要),不手动维护版本文件。

## 单一事实源边界(重要)

领域 PRD **只拥有**两类内容:① 需求条目(`P-XXX`);② 领域稳定 framing(角色 / 名词 / 背景 / 目标 / NFR)。
其余内容各有唯一 home,本文件**只链接、不复制正文**:

- 调研正文 → `project/research/`
- 产品设计、整体流程图、设计原则 → 同领域 `design.md`
- 视觉规范 → `project/ui-brand.md`
- 版本级内容(本次背景 / 范围 / 约束 / 验收 / 规划 / 评审) → `release-prd.md`,由 `/deliver` 生成

`/deliver` 生成交付物时,会从上述各 home 拉取并按 `release-prd-spec.md` 组装出完整文档 —— 所以这里不要把那些章节正文抄进来,否则就多出事实源。

## 条目编号规范

所有功能需求条目必须有**稳定编号**。编号规则和格式决定了 `/deliver` 能否自动识别版本变更,请严格遵守。

### 编号格式

- 固定前缀 `P-`,三位数字:`P-101`、`P-102`、`P-215`
- 编号**一旦分配永不变化**;重命名、搬移、重新排序都保留原编号
- 不同领域独立编号段,避免冲突(如用户管理用 `P-1xx`,订单用 `P-2xx`)
- 废弃条目保留编号并打标,**不删除**;仅删除会让版本 diff 失去锚点
- 条目内规则编号用 `R-<条目号>-NN`(如 P-108 的规则为 `R-108-01`),与条目关联、避免跨条目冲突

### 条目标题与状态

每个需求用二级标题 `## P-XXX <简短标题>` 作为段落起点(这是 `/deliver` 解析 diff 的锚点)。
标题下第一行必须是 `**状态**:` 行,内容格式:

- `新增@v1.x` — 该版本首次引入
- `修订@v1.x` — 该版本对原条目做了修改;可累加多版本,如 `新增@v1.0;修订@v1.2;修订@v1.3`
- `废弃@v1.x` — 从该版本起不再生效,但保留条目供追溯

### 条目内部结构

每个 `## P-XXX` 下用三级标题展开,对齐交付物规范 `release-prd-spec.md` §7「详细需求」的模块结构(标 `可选` 的按需裁剪):

````markdown
## P-101 新增角色审批流

**状态**: 新增@v1.0
**模块**: 角色管理
**优先级**: P0
**来源**: <调研编号 / 追加>

### 功能概述

2-3 句:这个条目做什么、给谁用、解决什么问题。

### 功能主流程(可选)

Mermaid 中文业务流程图 / 时序图;或链接 design.md 对应图,不重复绘制。

### 核心逻辑规则

规则表(复杂规则配场景举例):

| 规则编号 | 规则描述 | 触发条件 | 预期行为 | 备注 |
|----------|----------|----------|----------|------|
| R-101-01 | 一句话描述规则 | 什么情况触发 | 系统应该怎样 | 补充 |

或集合符号(权限类逻辑):

```
核心逻辑:操作人员.Games ⊇ 配置数据.Games
- 查看:操作人员.Games ∩ 配置数据.Games ≠ ∅
- 修改:操作人员.Games ⊇ 配置数据.Games
```

### 场景举例(复杂条目必填)

用具体数据(ID / 名称)说明规则的实际效果,覆盖边界与异常:

```
场景1:校验失败
- 操作人员权限:[20125, 20173]
- 配置数据 Games:[20125, 20082]
- 结果:拒绝 —— 缺少 20082 权限
```

### 页面与交互(可选)

只写关键入口与交互要点,UI 细节(布局 / 控件)链接 design.md,不在此重复。

### 数据列表 / 创建编辑字段(可选,仅业务属性)

| 字段标签 | 必填 | 选项来源 | 业务说明 |
|----------|------|----------|----------|
| FieldA | 是 | 组织列表 | 业务含义与约束 |

> 不写 camelCase 字段名、数据类型、控件类型 —— 这些在 design.md / 内部设计文档定义。

### 边界与异常

结构化列出边界、降级、并发、权限守卫等。

### 模块间依赖(可选)

与其他条目 / 领域的逻辑依赖。

### 验收标准

可被 QA 直接推导为用例的明确判定。
````
