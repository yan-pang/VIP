# 产品设计规范

> 默认用于在 `project/domains/<domain>/design.md` 中生成或完善 `功能详细描述`。  
> 进入本规范前，请先参考 `product-design-kit/design/design-init.md` 完成 `design.md` 的初始化内容，并同时阅读 `project/ui-brand.md`。

## 文档定位

- `design.md` 是领域主设计文档，也是默认唯一的详细设计文档。
- 多模块关系、共享规则、页面设计和复杂交互都直接写在同一份 `design.md` 中，不再额外拆独立设计文档。
- `project/ui-brand.md` 负责项目级默认设计规范，例如品牌色、字体层级、圆角、阴影、间距密度、组件气质和禁用样式。
- `design.md` 默认继承 `project/ui-brand.md`；只有领域特有的设计例外才写回 `design.md`。
- `project/research/` 用于沉淀跨领域的材料来源、会议增量、交付参考和待确认问题(按迭代版本组织),不替代详细设计正文。业务领域下不再单独维护 `research.md`。

## 核心设计原则

### 1. 项目品牌规范优先

- 页面设计默认继承 `project/ui-brand.md` 中已经确定的项目级规则。
- 不在 `design.md` 里重复维护品牌色、默认字体、默认圆角、默认密度等长期稳定规则。
- 如果某个页面必须偏离项目默认规范，单独写明“为什么要例外、例外范围是什么、会影响哪些交互或实现”。

### 2. 信息层级清晰

- 重要信息优先显示，次要信息折叠、分层或按需展开。
- 页面结构、按钮层级、状态提示和辅助说明要形成稳定的视觉路径。
- 多模块领域先写清“模块关系和职责”，再写“单页细节”，避免一开始就陷入局部页面。

### 3. 流程闭环完整

- 设计应覆盖主流程、异常流程、状态流转和关键边界场景，而不是只写成功路径。
- 列表、表单、弹窗、tab、步骤流都应明确进入条件、退出条件、成功反馈和失败反馈。
- 操作按钮和交互路径要由数据状态、权限状态或上下文状态驱动，而不是写成静态摆设。

### 4. 表达一致、可实现

- 页面、弹窗、表单、列表、状态表达和错误提示要保持一致，不要一页一个交互范式。
- 设计描述应能直接映射到页面、路由、Mock、状态和组件实现，不写无法落代码的抽象口号。
- 复杂页面优先使用“布局草图 + 结构表 + 交互表”的组合表达，而不是堆大段说明。

### 5. 响应适配与可访问性

- 关键功能在常见桌面尺寸下始终可见，移动或窄屏时要明确折叠、换行或重排策略。
- 设计中应考虑对比度、焦点态、键盘操作、错误提示可感知性等基础可访问性要求。
- 如果项目对密度、可点击面积或响应式断点有默认要求，统一继承 `project/ui-brand.md`。

## `design.md` 应该写什么

这份规范只负责 `design.md` 的 `功能详细描述`。  
进入详细设计后，正文至少覆盖以下内容：

1. 领域结构与模块关系
2. 页面清单
3. 业务流程图
4. 共享规则与状态边界
5. 页面详细设计与模块展开
6. 关键交互与边界场景
7. 项目品牌规范的例外说明（按需）

如果领域很复杂，也仍然优先在同一份 `design.md` 中展开，而不是继续拆出新的独立设计文档。

## `design.md` 不应该写什么

- 不要重复 `project/ui-brand.md` 中已经稳定的项目级视觉规则。
- 不要把原始会议纪要、聊天记录或需求原文大段堆进正文。
- 不要把 `project/tech/` 中的研发技术设计正文整体搬进来;只有会影响 demo 表现的稳定结论才回写。
- 不要把纯交付口径、测试素材或待确认材料直接写成详细设计正文。
- 不要只列一堆字段或按钮，而不解释它们的目标、触发条件和系统反馈。

## 多模块领域如何组织

当一个领域天然包含多个相互关联的页面或模块时，建议在 `design.md` 的 `功能详细描述` 中按下面顺序组织：

1. 先用一节写“领域结构与模块关系”
2. 再列出页面清单和入口关系
3. 再写共享规则、共享状态和数据边界
4. 最后按页面或模块逐项展开细节

建议至少补两张表：

### 领域结构与模块关系表

| 模块 / 页面 | 主要目标 | 入口 | 依赖对象 | 关联关系 | 备注 |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |

### 页面清单表

| 页面 / 模块 | 路径 | 主要角色 | 页面目标 | 主要功能区 | 备注 |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |

写法建议：

- 同一业务过程中的新增、查询、编辑、详情和批量操作，优先归到同一页面或同一组紧密关联页面中表达。
- 页面命名、模块命名、路由命名和代码中的叫法保持一致。
- 如果一个模块下只有一个主页面，不要为了“显得完整”再人为拆额外页面层级。

## 页面布局与功能区表达

复杂页面建议使用“概述 + 结构表 + 关键交互”的方式描述。  
不要优先写固定像素值；更重要的是区域职责、信息层级、相对位置和交互行为。

### 页面结构表

```text
页面标题 / 面包屑
查询筛选区
页面级操作区
主体内容区（列表 / 卡片 / 图表 / 详情）
分页 / 统计 / 辅助提示
```

也可以改用表格表达：

| 页面区域 | 页面区域细分 | 作用 | 主要元素 |
| --- | --- | --- | --- |
| 顶部区域 | 页面标题 / 面包屑 | 说明页面定位 | 标题、说明、返回入口 |
| 查询区 | 搜索 / 筛选 / 快捷条件 | 缩小数据范围 | 输入框、下拉、多选、时间范围 |
| 操作区 | 页面级按钮 | 承载主要动作 | 新建、导出、批量处理 |
| 主体内容区 | 列表 / 卡片 / 图表 / 详情 | 展示核心信息 | 表格、卡片、统计、详情块 |
| 辅助区 | 分页 / 统计 / 空态提示 | 补充上下文和反馈 | 分页、总数、说明文案 |

### 常见功能区的描述要求

#### 导航与菜单

当页面存在导航栏、左侧菜单或局部导航时，至少说明：

- 它服务哪个角色和场景
- 它和页面清单、路由、tab 的关系是什么
- 切换后是换路由、换内容区，还是只换局部状态
- 当前项、禁用项和无权限项如何表达

#### 查询与筛选

至少说明：

- 查询字段名称和 `prop`
- 匹配方式：精确 / 模糊 / 范围 / 多选
- 默认查询条件
- 控件类型和交互方式
- 搜索、重置、展开更多等按钮的触发行为

#### 页面级操作

至少说明：

- 按钮名称
- 显示 / 启用条件
- 交互行为和后续结果
- 成功 / 失败 / 无权限时的反馈方式

#### 主体列表 / 内容区

至少说明：

- 字段标签和展示优先级
- 数据来源和业务规则
- 是否支持排序、筛选、批量操作、行级操作
- 空状态、加载状态、错误状态如何表达

#### 表单

至少说明：

- 字段标签和数据字段名
- 数据类型和控件类型
- 必填要求和校验规则
- 默认值、提示信息、可编辑条件
- 选项来源和联动逻辑

#### 标签页

至少说明：

- 每个 tab 的定位和适用场景
- tab 切换是切换路由还是切换局部内容
- tab 间数据是否共享、是否联动
- 默认落点和返回策略

#### 弹窗 / 抽屉

至少说明：

- 触发条件和关闭方式
- 标题、主体内容和操作区的层级
- 字段、校验、异常处理和提交结果
- 如果存在多步骤或弹窗内子 tab，要说明数据保留和步骤切换规则

#### 卡片与数据展示

如果页面主体不是列表，而是卡片、看板或图表，至少说明：

- 每块卡片或图表回答什么问题
- 关键指标、图例、单位、时间范围和异常值如何表达
- 图表和列表、筛选、详情之间如何联动
- 哪些信息是主视觉，哪些信息应降级显示

## 交互与状态表达

### 1. 交互反馈

- 点击、提交、切换、批量处理等关键动作应在合理时间内给出反馈。
- 需要明确悬停、激活、聚焦、禁用等基础状态。
- 失败反馈要可定位、可重试，不能只给一句模糊错误。

### 2. 页面状态

至少检查：

- 空状态
- 加载状态
- 错误状态
- 权限受限状态
- 已选择 / 已过滤状态
- 历史数据兼容或极端数据量场景

### 3. 关键交互流程表

| 场景 | 触发 | 系统处理 | 成功结果 | 失败 / 异常 |
| --- | --- | --- | --- | --- |
|  |  |  |  |  |

如果一个页面存在明显复杂的事件流，也可以补充伪代码，例如：

```text
handleSearch -> 更新 filters -> page 归 1 -> 刷新列表
handleCreate -> 打开弹窗 -> 校验 -> 提交 -> 刷新列表
handleTabChange -> 更新当前子页 -> 保留必要上下文
```

## 绘图与表达规范

### 1. Mermaid

- 推荐使用 `graph TD` 或 `graph LR`。
- 节点命名尽量使用“动词 + 名词”，例如“提交订单”“校验权限”。
- 画图前建议先参考 `product-design-kit/coding/mermaid-v8.8.md`，避免使用不兼容写法。
- 如果流程较长，可拆成“业务流程”“页面流程”“状态流转”多张图。

### 2. ASCII 布局图

- 适用于页面结构、弹窗结构、多步骤流程和 tab 容器结构。
- 重点是区域层级、操作位置和信息分布，不要求精确像素稿。
- 复杂页面不要只写“主体内容区”，应尽量标明关键区域和主要操作。

### 3. 表格优先的写法

- 页面结构、查询字段、表单字段、表格列、弹窗内容、交互流程都优先用表格表达。
- 表格用于写清“有什么、何时出现、受什么约束”，短文字用于补目的和背景。
- 如果只靠自然语言会让人反复来回读，优先改成表格或列表。

## 页面详细设计要求

每个页面或模块章节至少应覆盖：

### 1. 页面概述

- 页面目标定位
- 主要角色
- 页面入口和出口
- 这个页面负责什么、不负责什么

### 2. 页面功能流程

- 用 Mermaid 描述主流程、异常流程和关键分支。
- 如果存在多角色差异、状态流转或多步骤流程，需要明确标出分支条件。

### 3. 数据流说明

- 输入：用户输入、外部数据、路由参数、上下文状态
- 处理：校验、状态计算、联动规则、接口交互、权限判断
- 输出：列表刷新、状态变化、结果提示、后续跳转

### 4. 页面布局设计详情

- 页面整体区域划分
- 查询 / 筛选区与操作区的位置
- 主体内容区的结构
- 分页、统计、辅助说明等附属区域

### 5. 功能区详情

建议至少补这些子表：

- 搜索区字段表
- 页面级操作表
- 表格 / 内容区字段表
- 弹窗 / 抽屉字段表
- 子 tab / 多步骤说明（按需）

### 6. 页面交互说明

- 谁触发
- 触发条件是什么
- 系统如何反馈
- 失败时如何处理
- 是否需要保持上下文状态

### 7. 边界场景

- 空状态
- 加载状态
- 错误状态
- 权限受限状态
- 撤销、取消、返回和中断恢复

### 8. 设计例外说明（按需）

只有在页面必须偏离 `project/ui-brand.md` 时才写：

- 例外点是什么
- 为什么必须例外
- 影响哪些页面区域或交互
- 是否需要同步调整实现

## 文档关系

- `project/ui-brand.md`:项目级默认设计规范,负责长期稳定的品牌与视觉规则。
- `project/domains/<domain>/design.md`:领域主设计文档,负责业务设计、页面设计、交互设计和例外说明。
- `product-design-kit/design/design-init.md`:负责 `design.md` 的初始化阶段。
- `project/research/`:项目级调研工作台,按迭代版本组织,承接跨领域的材料来源、会议增量、交付参考和待确认问题。
- `project/tech/`:项目级研发技术设计,按迭代版本组织。
- `project/domains/<domain>/delivery/prd.md`:领域长期 PRD(权威源,持续演进)。
- `project/delivery/v1.x/`:版本级对外交付产物,由 `/deliver` 从领域 PRD 组装。
- `project/overview.md`:负责维护跨领域状态、共享概念、跨领域约定和公共能力。
