# PRD 自查清单

> 来源：Sensors Platform Products 设计 Checklist，已根据当前项目（前端可视化原型）适配。
> 起始节点：领域已完成设计初始化，至少已经写完 `design.md` 中除 `功能详细描述` 外的部分。
> 结束节点：领域可以进入开发或对外交付。

## 使用时机

本清单分为 3 个关卡，对应流水线中的 3 个质量检查点：

| 关卡 | 时机 | 用途 |
|------|------|------|
| 关卡 0 | `design.md` 初始化部分基本完成后 | 确保需求理解到位、方案选型合理 |
| 关卡 1 | `design.md` 的 `功能详细描述` 完成后 | 检查主设计文档的完整性 |
| 关卡 2 | `project/domains/<domain>/delivery/` 文档完成后 | 评审前最终确认 |

使用方式：
- 每次做完对应环节后，逐项对照勾选
- 勾不上的地方就是需要补充或确认的地方
- `guide-agent`、阶段 skills 和人工自查都统一使用这份清单
- 若 `project/research/` 或 `project/tech/` 下存在相关版本文件,先确认其中需要进入交付物的参考点、技术约束和待确认项是否已经处理

## 关卡 0：需求理解确认

> 在正式开始产品设计前完成。确保你已经想清楚了"要做什么"和"为什么这样做"。

### 理解需求

- [ ] 需求背景是怎么样的，需求来自哪里
- [ ] 需求描述的核心问题是什么，原始需求是否表达清楚
- [ ] 是否需要跟进需求方进一步了解想法
- [ ] 是否是对现有功能的升级优化（如是，需确认现有逻辑）
- [ ] 提炼后的需求核心点在问题的定义上是否足够清楚（包括需求边界）
- [ ] 需求是否符合当前产品阶段（是否过于超前或已经过时）

### 方案设计

- [ ] 方案是不是最优解，是否还有其他解决方案
- [ ] 方案的优缺点是什么，为何要选择这种方案
  > 不同方案一定存在优劣势，如果自己难以判断，可以交给 leader 从更高视角判断
- [ ] 当前的业务、技术层能否支撑该方案（实施成本是否过高）
- [ ] 该方案对交付影响有多大（如涉及交付，需跟交付同学沟通评估）
- [ ] 方案是否涉及跨部门业务影响，是否需要跟其他业务部门确认
- [ ] 方案对整体产品矩阵的影响
  > 例如信息结构调整、功能入口变更等对其他产品线的影响
- [ ] 该方案是否还有优化空间
- [ ] 方案是一次性解决问题还是需要分期解决
- [ ] 是否存在类似竞品，是否完成竞品分析
  > 避免闭门造车，确保方案经过了充分的竞品调研
- [ ] 是否涉及版本兼容问题
- [ ] 是否涉及特殊客户/部署类型的兼容问题
- [ ] 是否存在不同产品线的差异化对待

### 跨模块依赖确认

- [ ] 本模块引用的外部模块是否已全部确认（如 PersonalGroup 被 Queue 引用）
- [ ] 未确认的依赖是否已标注"待确认"状态
- [ ] 依赖模块的接口定义（字段、行为）是否与本模块设计一致

### 方案确认

- [ ] leader 是否进行了确认
- [ ] 是否进行了组内确认
- [ ] 是否进行了组外部门确认（如需要）
- [ ] 是否需要高管确认（如需要）
- [ ] 方案的确认结果如何（完全通过 / 存在意见 / 问题较多涉及根本出发点）
- [ ] 方案确认过程中是否进行意见收集
- [ ] 是否需要二次确认（如争议较多，修改后再次确认）

## 关卡 1：产品设计自查

> 在产品设计（步骤 3）完成后、生成对外文档前进行。检查设计文档的完整性和质量。

### 背景描述

- [ ] 该需求要解决的核心问题描述是否清晰
- [ ] 背景描述是否有原始需求可引用
- [ ] 需求方是谁，使用场景是什么样的
- [ ] 是什么原因导致现在的需求得不到满足

### 影响范围

- [ ] 该需求会对哪些方面产生影响（内部/客户/代码层）
- [ ] 对客户的影响范围有多大
- [ ] 对产品矩阵的功能影响有多大（需与相关产品线 PM 确认）
- [ ] 对前后端影响有多大（需跟开发沟通评估）

### 方案描述

- [ ] 是否描述清楚核心解决方案
- [ ] 文字话术是否简单易懂
- [ ] 整体设计是否符合核心解决方案

### 流程图 — 任务流程图

- [ ] 是否需要流程图来描述整个设计方案的流程
  > 任务流程图需要确定开始和结束节点
- [ ] 流程图元素的使用是否正确
- [ ] 整体流程的覆盖面是否足够全
- [ ] 是否存在遗漏的流程节点
- [ ] 是否考虑到特殊的流程节点（特殊节点需标注）
- [ ] 流程图是否美观易读

### 流程图 — 业务流程图

- [ ] 是否考虑到该方案会涉及到哪些产品线/业务
- [ ] 在各产品线/业务中的流转是否正确
- [ ] 是否存在异常流程，异常流程怎么处理

### 流程图 — 页面流程图

- [ ] 流程是否完善，有无页面缺失
- [ ] 是否有异常流程，怎么处理（异常流程需在正常流程旁标出）
- [ ] 流程是否合理（如逆向操作的页面跳转是否合理）

### 整体交互

- [ ] 页面跳转是否完整（从起始点到结束的完整操作流程）
- [ ] 页面元素是否完整
- [ ] 页面的逆向操作是否有完整路径，返回是否合理
- [ ] 是否需要动效交互辅助说明（对于语言描述吃力的功能点）

### 文案

- [ ] 页面的文案是否易理解，是否存在歧义，是否有错别字
- [ ] 文案是否符合整体设计的文案规则

### 数据内容

- [ ] 展示的数据是否考虑了真实数据量级
- [ ] 是否考虑了页面数据正常和异常的展示
- [ ] 是否考虑到极端情况的展示样式（如数字位数过多）
- [ ] 数据是否涉及排序问题
- [ ] 数据展示是否涉及分页
- [ ] 是否考虑了数据量限制的问题

### 特殊情况

- [ ] 页面的交互是否符合全局系统交互范式
- [ ] 页面操作中是否考虑了终止操作的情况（如编辑取消后内容是否保存）
- [ ] 多业务状态下是否考虑了兼容问题（需确认现有逻辑）
- [ ] 是否考虑了无网络时的情况
- [ ] 是否存在版本兼容的设计
- [ ] 是否需要用户操作提示

### 逻辑处理 — 增加

- [ ] 点击增加后的提示信息是什么
- [ ] 增加的内容信息是否是必填字段
- [ ] 增加是否有限制条件（字数、字符、格式）
- [ ] 哪些情况不允许增加（如权限限制）
- [ ] 增加过程中输入的参数有哪些
- [ ] 是否支持批量增加
- [ ] 操作提示有哪些
- [ ] 增加是否进入到编辑状态
- [ ] 编辑状态是否支持删除
- [ ] 确认提交后返回到哪里
- [ ] 是否支持连续添加
- [ ] 增加的类型有哪些（文本、图片、视频等）
- [ ] 增加后的数据如何处理存放（需确认关联的现有逻辑）

### 逻辑处理 — 删除

- [ ] 点击删除后是否需要二次确认
- [ ] 删除后是否可以快速撤销
- [ ] 删除中断时（如断网），被删除内容如何处理
- [ ] 删除后是否可恢复，如何操作
- [ ] 是否支持批量删除，如何操作，如何提示
- [ ] 在增加的过程中是否支持删除
- [ ] 删除操作后数据如何处理

### 逻辑处理 — 修改

- [ ] 修改的限制条件是什么
- [ ] 哪些参数可以修改，哪些不可以修改
- [ ] 是否支持批量修改
- [ ] 修改过程中如何取消修改
- [ ] 修改后可以撤销吗
- [ ] 修改后提交需要二次确认吗
- [ ] 修改后的数据如何处理

### 逻辑处理 — 查询

- [ ] 查询的入口在哪里
- [ ] 查询时出现异常情况怎么显示
- [ ] 查询的对象是什么
- [ ] 哪些字段支持查询
- [ ] 查询条件是什么（按类别、时间等）
- [ ] 查询的方式是什么（搜索/筛选）
- [ ] 搜索条件是什么（ID、用户名、手机号等）
- [ ] 匹配方式是什么（精准匹配/模糊匹配）
- [ ] 查询结果有屏蔽要求吗，对查询结果可以进行哪些操作
- [ ] 查找结果怎么展示、怎么排序、没找到怎么显示
- [ ] 打开结果展示的方式（新标签页/当前页/弹窗/详情页），是否支持进一步跳转

### 逻辑处理 — 显示

- [ ] 敏感数据如何显示，怎么配置（隐藏、权限设置）
- [ ] 显示内容的优先级（必要字段、排版、呈现方式）
- [ ] 显示的层级关系（父子级嵌套关系）
- [ ] 功能操作的方式、操作过程展示、操作结果展示如何呈现
- [ ] 页面的默认选中焦点、选中项、默认值展示
- [ ] 页面显示元素的交互状态有哪些（悬停、点击、选中、禁止）
- [ ] 显示的顺序（创建时间、修改时间、类别）
- [ ] 列表显示的样式、一页显示的数量、分页显示数量
- [ ] 列表是否支持快捷操作（排序、筛选、搜索）
- [ ] 显示元素的数量范围、文本过多如何显示
- [ ] 内容为空怎么显示
- [ ] 错误提示怎么显示，显示内容是什么
- [ ] 哪些内容是前端固定的，哪些内容是服务端返回的（需确认现有逻辑）

### 逻辑处理 — 计算

- [ ] 多久计算一次
- [ ] 哪些数据参与计算
- [ ] 参数的限制有哪些
- [ ] 数据变化的规则（实时更新/自动拉取/推送/T+1 更新）
- [ ] 哪些信息需要默认保存、自动填充
- [ ] 需要什么统计方法

### 逻辑处理 — 传输

- [ ] 传输内容格式支持哪些（文本、图片、视频、数据）
- [ ] 哪些需要传，哪些不需要传
- [ ] 手动传还是自动传
- [ ] 传输的内容是什么
- [ ] 上传文件是否有格式限制、大小限制
- [ ] 上传文件后是否支持显示文件名
- [ ] 上传文件是否要显示格式信息、格式的提示
- [ ] 上传后是否允许重复上传、覆盖上传、取消上传
- [ ] 是否可以批量上传，批量上传后如何显示
- [ ] 上传后是否可以删除，如何删除

### 逻辑处理 — 异常情况

- [ ] 数据异常如何处理
- [ ] 异常处理：为空、字符超长、超时
- [ ] 网络异常数据如何处理，如何提示用户（中断/请求刷新/重试/关闭）
- [ ] 账号异常情况（登录、多地登录）
- [ ] 中断、失败、超出限制如何处理
- [ ] 敏感词、内容违规如何处理
- [ ] 无法获取数据时的处理
- [ ] 加载时间过长（页面多久超时、超时后提示方式）
- [ ] 状态码提示（404、403）

### 逻辑处理 — 权限

- [ ] 是否存在不同权限的处理问题（需确认现有逻辑）
- [ ] 不同权限的增删改查处理结果是否相同
- [ ] 不同权限的页面展示是否存在差异性

## 关卡 2：评审前自查

> 在对外需求文档（步骤 6）完成后、正式评审前进行。确保文档已准备好对外交付。

### 整体需求评估

- [ ] 有哪些计划解决的问题需要评估
- [ ] 评估的方法有哪些（数据/访谈）
- [ ] 评估人是谁
- [ ] 评估时间点
- [ ] 评估结果如何
- [ ] `tech/` 中的关键接口、依赖、异常处理和兼容性约束是否已经进入 PRD 或测试文档

### 国际化（如需要）

- [ ] 是否需要实现国际化
- [ ] 新增名词有哪些
- [ ] 整体界面简繁体、英文的翻译是否遗漏

### 外部确认

- [ ] 是否需要跟外部部门确认需求
- [ ] 外部部门的确认情况如何

### 会议反馈

- [ ] 会议的反馈内容是否记录
- [ ] 是否对会议的反馈内容进行了处理

### 合规性

- [ ] 是否存在法律及合规性风险
- [ ] 哪些地方涉及合规性问题
- [ ] 涉及合规性问题如何处理
