# React 列表页开发方法

> 适用于当前 starter 的主流页面类型：搜索 + 表格 + 分页 + 弹窗。

如果你现在准备开始写页面，先记住这 5 条：

1. 简单列表页优先复用 `TablePageLayout`
2. 页面负责数据、状态和事件，通用组件负责结构和骨架
3. 搜索项、表格列、按钮尽量写成配置
4. 新页面至少同时检查 `views`、`router`、`services`
5. 做完后跑 `npm run check`

另外补一条默认前提：

- 页面视觉层级、信息密度和组件气质默认继承 `project/ui-brand.md`，只有 `design.md` 明确说明例外时才偏离。

记住这 5 条，这个 starter 就已经够用了。

## 最常见的页面类型怎么做

### 简单列表页

最推荐的做法就是：

- 页面里组织数据和事件
- 用 `TablePageLayout` 搭整体结构
- 用 `SearchBar` 渲染搜索项
- 用 `DataTable` 渲染表格
- 用 `DialogWrapper` 做新建 / 编辑弹窗

如果你的页面本质上是“搜索 + 表格 + 弹窗”，优先按这个套路来。

### 复杂筛选列表页

如果筛选很多，starter 默认没有现成的侧栏方案。此时可以：

- 页面层自己多包一层结构
- 但搜索项、表格、弹窗仍然优先复用通用组件

### 标签页或多子页模块

这类页面更适合用路由拆分：

- 页面外层做容器
- 子页面走 `Outlet`
- 菜单和路由一起注册

## 页面里应该放什么

页面组件主要负责这些事：

- 取数据或组织数据
- 管理状态
- 定义搜索配置、表格配置、按钮配置
- 处理业务事件

通用组件主要负责：

- 页面结构
- 通用交互骨架
- 通用样式和行为

简单说：

业务判断放页面里，通用展示放组件里。

## 一个列表页常见会有哪些状态

通常至少会有这些：

- `loading`
- `filters`
- `page`
- `pageSize`
- `tableData`
- `dialogOpen` 或 `drawerOpen`
- 当前编辑对象或当前选中对象

如果一个页面已经有很多状态，说明它可能需要补技术设计了。

## 为什么建议“配置优先”

因为这样更好改，也更容易和设计文档对应。

建议优先整理这几类配置：

- `searchItems`
- `actionButtons`
- `tableColumns`
- `tableActions`

不要一开始就把按钮逻辑和单元格逻辑散在一大堆 JSX 里，否则后面会越来越难改。

## 新页面最少要同步哪些位置

1. `src/views/`：页面文件
2. `src/router/index.tsx`：路由注册
3. `src/components/layouts/NavigationLayout.tsx`：菜单调整
4. `src/services/`：mock 数据补齐

少看一个地方，就很容易出现“代码写了但页面打不开”。

## mock 数据怎么准备

至少要能支撑这些演示：

- 主流程
- 空状态
- 关键异常状态

如果一个页面靠状态切换体现价值，建议准备 2 到 3 类不同数据，不要只放一组“永远正常”的 mock。

## 什么时候再抽共享组件

满足下面任意两条，再考虑抽：

- 两个以上页面都会复用
- 交互已经稳定
- 页面里出现明显重复
- 配置接口已经能抽清楚

如果只是某个页面暂时复杂，还没稳定，先留在页面内部更稳妥。

## 做完后的最小自查

- 路由能打开
- 搜索、分页、弹窗或关键交互能走通
- 文案和设计一致
- mock 数据能支撑演示
- `npm run check` 通过
