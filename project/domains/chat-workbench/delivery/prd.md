## 领域 framing(稳定信息)

### 名词解释

| 名词 | 解释 |
| --- | --- |
| ChatFlow | 本产品代号,VIP 私域运营客服后台 |
| 工作台(`/workbench`) | 客服接待主界面:聚合多号会话、收发消息、会话管理 |
| 控制台(`/control`) | 企微号运维界面:云桌面投屏、扫码登录、重启 / 删除号 |
| 无影云桌面 | 承载企微客户端的云桌面;每个企微号固定绑定一台,长期同 IP 登录规避封号 |
| RPA | 阿里云 RPA SDK,在云桌面里代为执行企微发送等操作 |
| 企微会话存档 API | 消息**接收**的唯一数据源 |
| 会话(conversation) | 同一(企微号 × 玩家)的持久会话;`conversationId` 跨轮次不变 |
| 轮次(round) | 一次"排队→会话→结束"的生命周期;同一会话可多轮,消息跨轮累积 |
| 指派人(assignee) | 当前负责该会话的客服;= null ↔ 排队中,非 null 且未结束 ↔ 会话中 |
| 主动发起态 | 已结束会话临时解锁输入区的 UI 态,不持久化 |
| 落库 | 会话至少有一条发送成功(`sent`)的客服消息,真实链路由会话存档 API 承担 |

### 领域背景

VIP 私域运营通过多个企业微信号触达玩家,客服需逐号登录企微、分散回复,人工操作成本高、难以聚合管理,且面临违禁词外发与企微号封禁的合规风控压力。ChatFlow 通过"云桌面托管企微号 + 会话存档接收 + RPA 发送 + 多号聚合工作台"把分散的接待收敛到统一后台。方案选型与合规风控约束见 `project/research/v1.0-qiwei-rpa.md`。

### 核心目标

| 面向的用户 | 解决的核心问题 | 目标结果 |
| --- | --- | --- |
| 客服(坐席) | 多企微号分散接待、逐号登录回复成本高 | 单工作台聚合所有有权限号的会话,统一收发与会话管理 |
| 客服 / 运营 | 违禁内容外发、企微号封禁风险 | 出站违禁词拦截 + 号状态监控告警 + IP 一致性规避封号 |
| 运营负责人 | 接待协作与分流不透明 | 会话指派 / 转接 / 结束的明确状态机与协作闭环 |

### 长期稳定角色

| 角色 | 职责定位 | 备注 |
| --- | --- | --- |
| 客服(坐席) | 登录 ChatFlow 接待玩家;一人可同时负责多个企微号;收发消息、管理会话、扫码登录与人工介入 | 本领域唯一交互角色 |
| 玩家 | 被 VIP 运营添加的企微好友;不登录 ChatFlow | 仅作为会话对端,不是系统用户 |
| 管理员 | 授权客服可见企微号、维护企微号-云桌面绑定 | 能力在 permission / ops-admin 领域,本领域只消费 |

### 非功能性需求(NFR)

| 选项 | 是否需要 | 说明/负责人 |
| --- | --- | --- |
| UI | 需要 | 详见 `design.md`(D1+D2+D3)与 demo |
| 权限 | 需要 | 客服账号 / 角色 / 企微号授权由 permission 领域提供,本领域消费 |
| 国际化 | 待定 | V1 中文优先 |
| 数据打点 | 待定 | 后续版本明确 |
| 安全合规 | 需要 | 出站违禁词拦截、IP 一致性、固定设备登录、录屏审计;详见 `project/research/v1.0-qiwei-rpa.md` |
| 性能测试 | 需要 | 关键时延:消息接收前端可见 ≤10s、发送送达判定 ≤5s、状态变更可见 ≤10s |
| 版本兼容 | 待定 | 真实链路接入契约后续在 `project/tech/` 补 |

### 非目标(V1 不做)

- 自动撤回(撤回为跳控制台手动引导,见 P-117)
- 自动结束会话、基于时间的自动状态流转(见 P-108 / P-113)
- 消息接收前端轮询兜底(见 P-104)
- 表情包 / 语音 / 卡片发送(见 P-114)
- 智能路由、自动分配、下班自动回复(automation 领域,v1.1+)

## P-101 企微号扫码登录

**状态**: 新增@v1.0
**模块**: 控制台(`/control`)
**优先级**: P0
**来源**: R018

### 功能概述

客服在控制台的云桌面视频流里完成企微号扫码登录,支持"新增号"与"已有号重连"两个入口;登录成功后号置为在线,投屏持续保留供后续运维操作。

### 核心逻辑规则

| 规则编号 | 规则描述 | 触发条件 | 预期行为 | 备注 |
|----------|----------|----------|----------|------|
| R-101-01 | 扫码登录入口 | 点"+ 添加号"(新增)或控制条"重新登录"(重连) | 后端请求无影云开启云桌面并打开企微客户端,主体投屏展示完整云桌面 | 非单独二维码图 |
| R-101-02 | 登录成功置在线 | 手机企业微信扫码确认 | 该号状态置"在线",左列卡片状态点变绿,≤10s 进入左列 | |
| R-101-03 | 投屏持续保留 | 登录成功后 | 投屏保留,供后续安全弹窗 / 绑定提示 / 人工撤回在同一区域操作 | |
| R-101-04 | IP 一致性规避封号 | 号长期登录 | 固定绑定一台无影云桌面、长期同一 IP 登录(后端保证,客服无感) | |

### 页面与交互

入口:`/control` 左列底部"+ 添加号"主按钮 / 控制条"⟲ 重新登录"(号卡片不提供右键菜单)。扫码态在投屏上叠加半透明遮罩 + 引导文案"请用手机企业微信扫描屏幕中的二维码"。详见 design.md 图 3、页面 2(`/control`)2.5.3 投屏区。

### 边界与异常

- 扫码超时(2 分钟内未扫)→ 投屏区显示重试提示。
- 已在其他云桌面登录冲突 → 视频流内出现企微原生提示 + 控制台叠加横幅"该号已在其他云桌面登录,请先退出" + "前往原桌面"按钮。
- 无影云连接失败 → 投屏占位骨架 + "云桌面连接中…" → 超时降级错误占位 + 重试。

### 验收标准

- 从"+ 添加号"发起,云桌面投屏在设定超时内出现可扫二维码;手机扫码确认后号状态在 ≤ 10s 内变为在线并进入左列。
- 登录冲突场景能看到冲突横幅与处置入口,不会静默失败。
- 重新登录复用同一号卡片,不产生重复号记录。

## P-102 多号同窗口切换

**状态**: 新增@v1.0
**模块**: 控制台 / 工作台
**优先级**: P1
**来源**: R019

### 功能概述

控制台与工作台对"多号"的语义不同:控制台一次只投屏一个号,工作台默认聚合所有有权限号的会话并支持按号筛选,客服无需强制选号即可工作。

### 核心逻辑规则

| 规则编号 | 规则描述 | 触发条件 | 预期行为 | 备注 |
|----------|----------|----------|----------|------|
| R-102-01 | 控制台切号语义 | 点击 `/control` 左列号卡片 | 切换主体投屏 = 看哪个号的云桌面;一次只投屏一个号 | |
| R-102-02 | 工作台按号筛选语义 | `/workbench` 左列底部"按号筛选"(多选) | 决定看哪些号的会话;默认勾选全部 = 不筛选 | |
| R-102-03 | 默认聚合 | 工作台进入 | 默认聚合所有有权限号的会话,不强制选号即可工作 | |

### 页面与交互

工作台:左列底部漏斗按钮(`FilterOutlined`)弹出向上的多选 Popover,顶部"全部号"半选 / 全选,各号带状态点;非全选时按钮主色高亮 + 右上角数字角标。控制台:点击左列号卡片切换主体投屏;当前选中卡片左侧主色实线 + 浅背景。详见 design.md 1.5.2、2.5.2。

### 边界与异常

- 工作台筛选取消勾选当前打开会话所属的号 → 直接关闭中列(不弹二次确认),回默认欢迎态。
- 工作台全部取消勾选 → 会话列表为空 + Tooltip"当前未勾选任何号,会话列表为空"。
- 控制台切号 → 停旧视频流、起新视频流;新流建连失败 → 占位错误 + 重试。

### 验收标准

- 工作台默认不选号即展示全部有权限号的会话;按号筛选后列表与角标数量一致。
- 控制台切号后主体投屏在合理时延内切换为目标号画面。

## P-103 企微号状态监控与告警

**状态**: 新增@v1.0
**模块**: 全局告警区 / 控制台
**优先级**: P0
**来源**: R020

### 功能概述

实时监控企微号在线 / 掉线 / 封禁三态,通过工作台全局徽章与控制台单号详情两层呈现;掉线 / 封禁直接影响接待并给出处置提示。

### 核心逻辑规则

| 规则编号 | 规则描述 | 触发条件 | 预期行为 | 备注 |
|----------|----------|----------|----------|------|
| R-103-01 | 三态监控 | 无影云状态事件走后端长连推送 | 号状态在线 / 掉线 / 封禁,状态变更前端可见 ≤10s | |
| R-103-02 | 两层告警并存 | 任意号告警 | 工作台顶栏铃铛徽章(全局)+ 控制台主体详细监控(单号)同步 | |
| R-103-03 | 掉线 / 封禁影响接待 | 号掉线或封禁 | 该号下所有会话标题栏标记离线 / 封号、输入区禁用;封禁顶部红色横幅持续显示至管理员处理 | |

### 页面与交互

顶栏铃铛:红点指示告警,hover 看最近 5 条,点击展开通知中心。控制台左列号卡片状态点(●绿 / ●灰 / ●红)+ 主体投屏实时画面。告警横幅采用领域设计例外的"风控告警大色块"(封号红 `#FF4D4F`/`#FFF1F0`)。

### 边界与异常

- 长连断开 → 顶部黄色横幅"实时消息暂时不可用,正在重连…" + 立即重试;期间状态保留最后同步值,重连后增量补齐。
- 多 Tab 同账号 → 状态以后端推送为准,< 2s 延迟同步。

### 验收标准

- 制造掉线 / 封禁后,顶栏徽章与对应号会话标题栏在 ≤ 10s 内反映新状态,且输入区被禁用。
- 长连断开期间顶部出现重连横幅,恢复后告警状态自动校正。

## P-104 消息接收

**状态**: 新增@v1.0
**模块**: 会话中心
**优先级**: P0
**来源**: R021

### 功能概述

以企微会话存档 API 为唯一数据源接收玩家消息并实时刷新 UI;玩家消息完整展示原文、不做任何过滤。本条是未读、聚合、分组、发送等事项的前置依赖。

### 核心逻辑规则

| 规则编号 | 规则描述 | 触发条件 | 预期行为 | 备注 |
|----------|----------|----------|----------|------|
| R-104-01 | 唯一数据源 | 玩家消息到达 | 经企微会话存档 API → 后端长连推送 → 前端,入库后实时刷新,前端可见 ≤10s | V1 不做前端轮询兜底 |
| R-104-02 | 入站不过滤 | 玩家发来消息 | 不做任何违禁词 / 脱敏过滤,完整展示原文(保留真实上下文) | |
| R-104-03 | 类型支持 | 收到各类型消息 | 文本 / 图片 / 视频 / 文件 / 链接 / Emoji 收发支持;语音 / 红包 / 位置 / 名片仅占位展示 | 清单见 design.md 共享规则 §5 |

### 页面与交互

中列消息流时间正序(老上新下),自动滚到底;玩家消息靠左白底 + 1px 边框。

### 边界与异常

- 消息流首次加载 → 整个中列骨架屏,3s 超时降级 spinner;后续增量拉取无 loading。
- 长连断开期间消息不丢,重连后增量补齐。
- 不支持类型(语音 / 红包等)→ 占位条提示,不阻塞其他消息渲染。

### 验收标准

- 玩家新消息在 ≤ 10s 内出现在对应会话消息流并刷新左列预览 / 未读。
- 玩家侧违禁内容原样展示,不被拦截或脱敏。

## P-108 会话分组与状态机

**状态**: 新增@v1.0
**模块**: 会话中心
**优先级**: P0
**来源**: R028

### 功能概述

会话状态机为排队中 / 会话中 / 已结束三态,由明确的驱动事件在状态间跃迁;左列在此基础上把"会话中"按指派人拆成两个分组展示 —— **会话中**(仅指派给当前客服)与**他人接待中**(指派人是其他客服,只读)。同一(企微号×玩家)跨轮次复用同一 `conversationId`,消息累积。

### 功能主流程

状态机详见 design.md 图 1。

### 核心逻辑规则

| 规则编号 | 规则描述 | 触发条件 | 预期行为 | 备注 |
|----------|----------|----------|----------|------|
| R-108-01 | 三状态互斥 | 任意时刻 | 会话状态只属于 排队中 / 会话中 / 已结束 之一 | 分组是展示派生,不是第四种状态 |
| R-108-02 | 分组顺序与折叠 | 渲染左列 | 固定顺序 排队中 → 会话中 → 他人接待中 → 已结束;各组可独立折叠(默认**他人接待中 / 已结束折叠**) | |
| R-108-11 | 会话中按指派人拆分 | 进行中会话渲染 | `active && 指派给当前客服` → 会话中(可回复);`active && 指派给他人` → 他人接待中(只读,不放进会话中,避免堆积干扰、漏回) | 由 `status + assigneeId` 派生 |
| R-108-03 | 进入排队中 | 玩家首条消息到达 | 置排队中,指派人为空 | |
| R-108-04 | 排队中 → 会话中 | 客服显式指派(含指派自己) | 置会话中 | 见 P-109;**已取消发消息隐式指派** |
| R-108-05 | 会话中 → 会话中 | 转接(指派人变更) | 状态不变,指派人变更 | 见 P-112 |
| R-108-06 | 会话中 → 已结束 | 客服点结束 | 移入已结束 | 见 P-113 |
| R-108-07 | 已结束 → 排队中 | 玩家再次发消息 | 重回排队中,指派人清空 | |
| R-108-08 | 已结束 → 会话中 | 主动发起态首条消息发送成功 | 置会话中 | 见 P-120 |
| R-108-09 | 会话 ID 跨轮次不变 | 同一(企微号×玩家)重开 | 复用同一 `conversationId`,消息跨轮次累积 | |
| R-108-10 | 跃迁由事件驱动 | — | 所有跃迁有明确驱动事件;V1 不做基于时间的自动流转;分组跃迁由指派 / 结束驱动,不由点击会话驱动(仅查看不改分组) | |

### 场景举例

```
场景1:同一会话跨轮次复用 ID
- 玩家首条消息 → 会话 C-001 进入排队中(指派人=null)
- 客服点"指派"选自己 → C-001 进入会话中(指派人=张三);排队中不可直接回复
- 客服点结束 → C-001 进入已结束(指派人 张三 保留作历史)
- 次日玩家再来消息 → C-001 重回排队中(指派人清空),消息接在历史下方
- 结果:始终是同一个 conversationId=C-001,消息跨轮累积
```

### 页面与交互

左列分组标题带计数;点击标题折叠 / 展开;跨轮次在消息流插入系统分割条(本次会话已结束 / 玩家重新发起 / 客服主动发起)。历史轮次默认只渲染最新一轮,上滚自动加载更早轮 + sticky"收起"胶囊。详见 design.md 1.5.5。

### 边界与异常

- 已结束 → 玩家再来消息 → 重进排队中,指派人清空需重新指派。
- 打开排队中会话不渲染输入区(须先指派接入,见 P-109)。
- 会话被转接给他人后,从我的"会话中"移入"他人接待中"分组(只读)。

### 验收标准

- 各驱动事件触发后会话出现在正确分组;会话中只含指派给自己的会话,指派给他人的落入"他人接待中";`conversationId` 跨重开不变,历史消息保留可上滚查看。

## P-109 会话指派(显式,取消隐式)

**状态**: 变更@v1.0
**模块**: 会话中心
**优先级**: P0
**来源**: 追加(V1 闭环必需)

### 功能概述

会话通过**显式**(指派按钮,可指派给自己)唯一路径产生指派人;**V1 取消隐式指派**(发首条消息自动指派)——排队中会话输入区不渲染,客服必须先指派接入才解锁回复,避免多人同时回复串消息。非指派人 / 排队中未指派会话只能只读查看。

### 核心逻辑规则

| 规则编号 | 规则描述 | 触发条件 | 预期行为 | 备注 |
|----------|----------|----------|----------|------|
| R-109-01 | 显式指派 | 中列顶部"指派"按钮选客服(可含自己) | 指派人置为所选客服,会话进入会话中;指派给自己则输入区解锁 | 候选列表**仅在线客服**,离线不可指派 |
| R-109-02 | 取消隐式指派 | 打开排队中(未指派)会话 | 输入区不渲染,提示"点击「指派」接入(可指派给自己)后即可回复";不再有"发首条自动指派" | 替代原隐式指派 |
| R-109-03 | 唯一指派人 | 任意时刻 | 一个会话只有一个指派人;指派人=null ↔ 排队中,非 null 且未结束 ↔ 会话中 | |
| R-109-04 | 非指派人 / 未指派只读 | 打开非自己指派 或 排队中未指派的会话 | 输入区不渲染:排队中→"点「指派」接入…";他人指派→"已指派给 XX,你只能查看" | |
| R-109-05 | 点击不改指派 | 仅点击会话卡 | 打开查看,不改指派、不改分组 | |
| R-109-06 | 发送失败不回滚 | 会话中消息发送失败 | 仅标红 + hover 原因,状态 / 指派人不变;取消隐式指派后无首条失败回滚 | |

### 场景举例

```
场景1:排队中先接入才回复
- 会话 C-002 排队中(指派人=null),客服打开只能看,不显示输入框
- 客服点"指派"选自己 → C-002 进入会话中(指派人=自己),输入区解锁可回复
- 发首条即便失败,也只标红可重发,C-002 仍在会话中(不回滚)

场景2:并发抢同一会话
- 会话 C-003 排队中
- 客服 A、B 几乎同时点指派给自己 → 后端冲突错误"会话已被 A 指派"
```

### 页面与交互

指派 / 转接弹窗(D3 模块 4):会话信息只读 + 目标客服单选列表(**仅在线客服**;头像 / 姓名工号 / 会话量 / 在线点)+ 选填内部备注(≤200 字);指派模式自己排首标"推荐",一步接入。

### 边界与异常

- 列表无在线客服可指派 → "暂无可用客服",确认禁用。
- **离线客服不进候选列表**(不支持指派 / 转接给离线客服)。
- 并发抢同一会话 → 后端冲突错误"会话已被 XX 指派"。

### 验收标准

- 排队中会话不可直接回复,点"指派"选自己后进入会话中且指派人正确、输入区解锁。
- 会话中消息发送失败不改状态 / 指派人(无回滚)。
- 非指派人 / 未指派会话无法发消息 / 结束 / 改标记置顶。
