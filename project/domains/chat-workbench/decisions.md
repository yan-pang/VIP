## 2026-07-23 — 6.3 合并失败诊断流程 + 新增「回捞比对」送达判据

- **已送达采“乐观显示 + 事后回捞比对修正”，新增失败原因「回捞比对失败」**：发送经 RPA 在云桌面执行，**RPA 执行完成即先乐观标「已送达」**；系统随后从企微会话存档回捞该消息与发送内容比对，比对通过维持已送达，**只有超时窗内回捞不到 / 比对不一致才把已送达修正为失败（回捞比对失败）**、提示异常。**为什么**：RPA 只能确认“动作执行了”，是否真正进入玩家会话须以企微存档为准；但为不阻塞客服，先乐观显示成功、事后核实，异常才回退。**如何应用**：`design.md` §4 图2 先乐观已送达再回捞比对；`prd.md` 6.3(1) 状态机 `发送中 → 已送达(先乐观显示)` + 新增 `已送达 → 失败(事后回捞比对失败)`，R-115-01 改乐观显示 + 新增 R-115-06，R-116-02 失败详情原因纳入回捞比对失败。
- **回捞比对失败不自动重发**：RPA 已执行，重发有重复送达风险，故视为发送异常走失败详情 / 人工介入，由客服判断，不进「结果未知」自动重发通道。
- **6.3(2) 与原 6.7「失败诊断与人工介入流程」合并**：两图都在讲“失败按原因分流 + 可诊断类进失败详情 / 人工介入”，重复。合并为单张 6.3(2)（失败与异常分类 → 处理与人工介入），删除原 6.7，原 6.8「已结束重新联系与主动发起流程」顺延为 6.7。**为什么**：去重、单一事实源，避免两处口径漂移。

## 2026-07-23 — 彻底移除「待发送」状态（发送前不可用一律直接失败）

> 覆盖同日「消息重试边界修正」中"临时执行依赖不可用 → 待发送队列、依赖恢复后自动重试"的口径。产品决定不做本机排队自动重发，简化消息结果模型。

- **移除「待发送」消息结果状态**：消息结果状态机只剩 发送中 / 已送达 / 失败 / 历史已撤回（历史撤回 v1.1+）。**为什么**：待发送引入本机排队、跨刷新持久化、依赖恢复自动重试一整套复杂度，产品侧认为发送前不可用直接失败、由客服手动重发已足够，且模型更可预期、可验收。
- **所有发送前临时执行依赖不可用一律直接失败**：云电脑 / 机器人不可用、**RPA 任务积压 / 触发限流**（本轮明确其也走失败而非排队）都置失败、hover 显示原因，依赖恢复后由客服**手动重发**（新的发送意图，幂等防重复送达），系统不排队自动重发。**如何应用**：`design.md` §4 硬限制 vs 临时依赖、重试边界、混发流程、主动发起刷新保留规则，`prd.md` 6.2/6.3/6.6/6.8、P-103/114/115/117/118/120、附录 9.2.4/9.4 全部同步。
- **系统自动重试只保留一类**：发送后「结果未知」（网络异常 / 超时 / 无回执，已提交未拿到回执）时消息保持"发送中"、以同一幂等键自动重发。**为什么**：这类消息已发出且结果未知，靠幂等补发不会重复送达；与"发送前没发出去"性质不同。
- **临时草稿 / 重新联系首条刷新保留规则简化**：删除"首条进入待发送→本机持久化+自动重试"分支；首条只要未成功送达（失败 / 未发送）即未落库、刷新清除。**为什么**：无待发送态后不再有需要跨刷新续跑的排队消息。

## 2026-07-23 — 消息重试边界修正（自动重试仅限“结果未知”）

> 上一轮把“自动重试”错误收窄为只有「待发送（临时依赖不可用）」，且把明确失败写成“手动重发”，与真实机制不符。按“消息未真正送达且结果可安全重发”重新定边界，覆盖当日更早的重试边界口径。

- **自动重试判据 = “未真正送达且结果可安全重发”**，只有两类：① 发送前**临时执行依赖不可用**（云电脑 / 机器人暂不可用）→ 待发送队列，依赖恢复后自动重试；② 发送后**网络异常 / 返回超时等结果未知的瞬时异常** → 同一幂等键自动重试。**为什么**：这两类要么没发出、要么结果未知，靠幂等键补发不会重复送达。
- **有明确失败原因的发送一律不自动重试（终态）**：违禁词、玩家删好友、账号离线 / 停用 / 封禁、频率上限、IP 异常、企微明确拒绝等，按原因处理（重新输入 / 锁定待恢复 / 人工介入）；客服手动重发是**新的发送意图**，不是系统重试。**为什么**：明确失败重发结果不变或会造成风控 / 合规问题，不应由系统自动补发。
- **6.3 拆两图 + 降到业务口径**：原状态机把“按原因决定重试”硬塞进跃迁导致口径混乱。现拆为 (1) **消息结果状态**（只画客服在消息角标上看到的 待发送 / 发送中 / 已送达 / 失败 / 历史已撤回，配“状态→含义→客服要做什么”表；**不画** `失败 → 发送中`——手动重发是新发一条）；(2) **失败按原因分类 → 客服如何处理**（结果未知→系统自动重发；违禁词 / 删好友 / 账号离线 / 频率 IP / 企微明确拒绝→各自处置）。**为什么**：① 状态机描述“有哪些状态”、重试策略本质是“按原因分类”，混画必乱；② PRD 面向业务 / 产品 / 研发 / 测试，去掉幂等键、自循环等工程黑话，用“系统会不会自动重发、客服要不要重发、玩家收不收得到”表达可观察行为（`external-prd.md:78` 要求消息结果对象必须单独给结果状态机，故保留不删）。事实源 `design.md` §4 保留幂等键等技术表述（供 AI / demo 实现），业务口径只作用于对外 PRD。

## 2026-07-23 — 会话状态转换矩阵「临时草稿」口径收束

> 6.2 转换矩阵里临时草稿两行口径过松，与 `design.md` 既有精确保留规则不一致，收束如下（矩阵向事实源看齐）。

- **临时草稿明确为“非 3 态”的工作台前置态**：仅工作台可见，不入玩家中心 / 消息管理 / 搜索索引；矩阵目标状态单元格标注此边界，避免被误读成第 4 个正式状态。
- **草稿期负责人写“不写入（延迟到首条成功才指派）”**（覆盖原“当前客服（延迟落地）”）：草稿期指派人保持 `null`，与“已结束→重新联系”的“不写入”同套“成功才落地”语义；原措辞会让人以为草稿期已落负责人。**如何应用**：矩阵负责人列、`design.md` 图 1 要点统一为“延迟到首条成功才指派”。
- **失败结果拆分“待发”与“首条失败 / 未发送”**（覆盖原合并的“保留临时草稿”）：待发→本机持久化、刷新保留、依赖恢复自动重试；首条失败或未发送→未落库、刷新即清除。**为什么**：两者刷新保留行为相反，合并会抹掉差异，与 [[无历史临时会话刷新保留规则]] 不一致。

## 2026-07-23 — 长期 PRD 补强前的业务事实收口（覆盖同日及更早冲突口径）

> 为把长期 PRD 从"功能摘要"升级为研发 / 测试可直接执行的完整业务规格,先在 `design.md` 锁定并统一以下业务事实,再重生成 `delivery/prd.md`。本条覆盖此前与之冲突的口径。

- **消息撤回 V1 暂缓**（覆盖 2026-07-22 一致性整改"ChatFlow 内直接撤回"的当期可用口径）：工作台当前不提供任何撤回入口;后端回传的历史已撤回消息仍在原位以"消息已撤回"占位正常展示。**为什么**:服务端撤回契约(时间窗、归属、幂等、回执)尚未确定,先不放开前端入口以免产生无法真正生效的操作。**如何应用**:P-117 / R045 保留历史编号不复用,标记为暂缓;权限矩阵、roadmap 和任务记录同步标注暂缓;对外 PRD 不描述当前可用的撤回操作,也不写实现差异或开发任务。v1.1+ 契约明确后再恢复。
- **违禁词等硬限制 = 失败气泡 + 清空编辑区 + 不支持"重新编辑恢复原文"**（对齐仓库现有实现,覆盖此前"命中后保留草稿"口径）：文本命中违禁词时仅该文本项拒绝,生成一条 hover-only 失败气泡("命中游戏违禁词、未发送")并清空编辑区(不入待发队列);**不提供"重新编辑恢复原文"入口**,客服如需重发须重新输入;同批次附件继续独立评估。**为什么**:与 `WorkbenchPage.tsx` 现有行为一致(命中文本置 `status:'failed'`、`handleSend` 清空编辑区、附件独立发送,无 re-edit 恢复),沿用统一失败态比"输入框标红保留"更一致、可追溯;不引入 PLAN 初稿里的"重新编辑恢复原文"以免与实现和后续维护脱节。
- **会话标记改为个人单选固定三值**（覆盖此前"自定义 / 可多选标记"）：标记为个人视角单选,固定"跟进中 / 重要 / 待回访"三值,选新值替换旧值,可清除;仅当前负责人可改。**为什么**:收敛为固定语义避免标记泛滥,卡片第三行只展示单个图标,口径清晰可验收。
- **已读口径明确"实际可见后才已读"**：玩家消息只在页面前台、目标会话已打开且该消息在消息流中实际可见后,才对当前客服标记已读并扣减其未读;仅打开会话但消息仍在未加载 / 未滚动到的历史轮次中不算已读。**为什么**:避免"打开即清零"导致真实未读被误消。
- **无历史临时会话刷新保留规则**：首条成功→转正式会话并落库(刷新保留);首条进入待发送→本机持久化(刷新保留,依赖恢复自动重试);首条明确失败或未发送→未落库(刷新清除)。**为什么**:待发送需要跨刷新继续自动重试,失败 / 未发送不应残留脏临时会话。
- **好友关系异常不改变正式会话状态与分组**：`removed_by_agent` / `removed_by_player` 时只锁定发送、清空草稿附件、取消未完成的临时联系态,会话的排队中 / 会话中 / 已结束状态与左列分组不变;恢复 `normal` 后解锁,但不自动指派或重开。
- **账号级硬限制与临时执行依赖必须区分**：企微号离线 / 停用 / 封禁 = 账号级硬限制,禁止普通发送与"重新联系",恢复在线后重判;云电脑 / 机器人暂不可用 = 临时执行依赖,合规消息进入待发队列自动重试,不禁用"重新联系";IP 异常 / 频率上限 = 全局硬限制,禁止发送、不入待发。

## D1 决策记录(V1 阶段针对外部依赖的决定)

分三类:**锁值(后续不再改)**、**V1 临时决策(看反馈再微调)**、**V1 明确不做(v1.1+ 再考虑)**。

### V1 锁值

| 决策点 | 决定 | 影响范围 |
| --- | --- | --- |
| 消息接收机制 | 后端长连推送(WebSocket / SSE),前端不做轮询 | 消息接收 / 未读 / 状态同步的前端设计都按推送建模 |
| 企微扫码登录呈现 | 全屏视频流投屏完整云桌面 | 登录抽屉 UI 按视频容器设计,含控制条 |
| 实时性 SLA | 消息 ≤ 10s、发送 ≤ 5s、状态同步 ≤ 10s;超 2× 告警 | UI 文案、超时判定、顶部告警触发逻辑 |

### V1 临时决策(未来可能微调)

| 决策点 | V1 决定 | 未来可能的变更方向 |
| --- | --- | --- |
| 权限粒度 | 按**企微号级**最细粒度,由 permission 领域承担 | 若要按"游戏/团队"分组授权,在 permission 领域内扩展,本领域接口不变 |
| 客服账号体系 | permission 领域**自维护**(账号、角色、授权关系全在内) | 后续若接外部 SSO,由 permission 领域内兼容,不影响 chat-workbench |
| 已结束会话重新联系 | 深链进入时只读；客服显式点击“重新联系”后输入区解锁，但状态机 / 指派人**不在点击时变更**，首条发送成功才跃迁到“会话中”并指派当前客服；失败或待发保持已结束 | 可结合 player-center 补“重新联系前必读字段提醒”；沉淀稳定后与 v1.1+ 智能路由一起评估统一入口 |
| `/control` 控制台 UI | 取消"控制台顶部条"和"返回工作台"按钮(顶栏 tab 已可切换)。左侧只展示已配置且已授权的企微号，不提供独立登录入口；系统管理员或运营主管在当前号投屏内、RPA 下线后可接管并操作。关闭投屏不影响左侧号卡、企微登录或 RPA | "刷新视频流/全屏/跳工作台对应会话"等在反馈明确时再补 |

### V1 明确不做

| 决策点 | V1 做法 | 去向 |
| --- | --- | --- |
| 敏感信息脱敏 | 不做,所有位置看原文 | v1.1+ 运营侧出字段清单后再加渲染层 |
| 消息撤回 | 历史方案曾跳云桌面手动撤回 | 已由顶部决定覆盖为 ChatFlow 内直接撤回请求 |

### 继续跟进的外部对齐项(不阻塞 D2)

- **permission 领域内部细节**:角色定义、企微号授权语义、登录鉴权、转接候选范围(R031),由 permission 领域 design 定义。
- **撤回服务契约**：不再依赖云桌面；待生产确认撤回时间窗、幂等和最终回执。
- **敏感信息脱敏清单**:V1 不做,运营侧给字段清单后 v1.1+ 引入渲染层。
- **会话存档 API / 无影云 API 契约**:长连消息格式、视频流协议、事件类型集合等接入细节。
- **RPA 调研原有待确认问题**:昵称 vs userid 搜索、MVP 规模、CS 资源协调等,在 [`research/v1.0-qiwei-rpa.md`](../../research/v1.0-qiwei-rpa.md) 的「待确认问题」跟踪。
