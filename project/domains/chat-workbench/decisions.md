## 2026-07-22 — 跨域一致性整改（覆盖同日及更早冲突口径）

- RPA、桌面或企微客户端暂时不可用时，已通过硬风控校验的消息进入**可见待发队列**，标记未送达并在依赖恢复后自动重试；禁用 / 封禁、IP 漂移、违禁词、频率硬上限等风险属于硬拦截，不入队且任何角色都不能绕过。
- 已结束会话从搜索、玩家详情或深链进入时始终只读；客服必须显式点“重新联系”，首条消息成功后才重新进入会话中。好友关系为 `removed_by_agent` 或 `removed_by_player` 时均禁止发送，恢复 `normal` 后也不自动重开或指派。
- 从玩家中心对无历史会话主动发起时，首次成功发送前只创建 Workbench 内的临时会话草稿；成功后才成为正式会话并被玩家中心、消息管理与搜索索引。失败或待发时保留临时态以便重试。
- 消息撤回改为 ChatFlow 内直接请求：普通客服仅能撤回本人消息，运营主管和系统管理员可撤回授权范围内团队消息；服务端仍需校验时间窗、消息归属、幂等与最终结果。
- 术语拆分：资源上的长生命周期执行单元称“机器人常驻任务”；一次编辑提交称“消息发送批次”，批次内每条内容有独立结果，不再把两者都叫 RPA 任务。

## 2026-07-22 — V1 Mock 实现审计快照
- 工作台已覆盖会话四分组、显式指派 / 转接、文本与附件混发、违禁词逐条拦截、失败详情与控制台介入、主动发起、综合搜索和玩家档案侧栏；当前实现使用前端共享 Mock Store，刷新恢复种子数据。
- 工作台与控制台路由切换时，SPA 会话内保留会话 / 消息变更、当前会话、企微号筛选、主动发起状态、草稿附件与消息历史浏览位置；浏览器刷新后的持久化仍只覆盖已成功落库的主动发起会话。
- 综合搜索改为订阅工作台 SPA 运行态，不再只查询静态种子，因此本次运行中新收 / 新发的消息和主动发起会话可立即检索。联系人按关系备注与昵称匹配，标签按当前账号权限统计关联玩家并跳转玩家筛选页；消息结果携带 `conversationId + messageId`，工作台自动展开目标历史轮次、滚动定位并短暂高亮。
- 搜索交互按稳定设计补齐 300ms 防抖、单字检索、全部 / 联系人 / 消息 / 标签四分类、命中词高亮、`Tab` 切分类、上下键选项与 `Enter` 跳转。真实浏览器已回归动态消息、标签跳转和跨轮次消息定位，控制台无 error / warning。
- 真实上线仍依赖会话存档长连、机器人常驻任务、发送批次逐条结果、无影云视频流、服务端会话状态机及 SCRM 数据契约；当前结果和录屏仅用于 Mock 演示。

## 2026-07-22 — Permission 最终验收联动
- 系统管理员与运营主管可在自身 `visibleAccountIds` 范围转接进行中的会话，即使该会话当前由其他客服接待；消息输入、标记、置顶和结束会话仍只对当前负责人开放。普通客服只能转接自己负责的会话。
- 转接候选排除会话当前负责人而非操作人，因此管理员或主管重新调度他人会话时可以选择自己接管；提交层同步校验操作权限，不能依赖按钮隐藏。
- 失败详情改为绑定失败消息所属的运行时会话，不再从静态种子反查；“人工介入”是否展示也按该失败会话的企微号权限判断，避免切换当前会话后发生错位。

## 2026-07-22 — 玩家好友关系恢复联动
- 好友关系以 player-center 的 `(playerId, accountId).relationStatus` 为单一权威源，工作台只维护实时投影。企微侧返回任一删好友状态时，同组合的会话清空草稿、锁输入并取消未发送的主动发起 / 重新联系临时态；不是只在失败消息旁显示提示。
- 玩家后续重新加回企微号、关系恢复为 `normal` 时，工作台立即解除删好友横幅和锁定，但不自动恢复已结束会话、指派客服或绕过企微号离线 / 封禁、他人接待等其他权限与状态限制。

## 2026-07-22 — 企微混发结果口径
- 覆盖 2026-07-17 的“合并为一条图文气泡”决定：一次提交形成一个消息发送批次，文本和每个附件拆成独立消息并共享 `sendBatchId`；旧记录的 `rpaTaskId` 只作兼容读取。
- 一个混发任务允许部分失败，只在失败消息旁展示统一红色失败图标和 hover 原因，不增加批次级复杂状态。
- 违禁词只校验单条文本内容；文本命中时仅文本消息失败，同任务内附件继续发送。

## D1 决策记录(V1 阶段针对外部依赖的决定)

分三类:**锁值(后续不再改)**、**V1 临时决策(看反馈再微调)**、**V1 明确不做(v1.1+ 再考虑)**。

### V1 锁值

| 决策点 | 决定 | 影响范围 |
| --- | --- | --- |
| 消息接收机制 | 后端长连推送(WebSocket / SSE),前端不做轮询 | 消息接收 / 未读 / 状态同步的前端设计都按推送建模 |
| 企微扫码登录呈现 | 全屏视频流投屏完整云桌面 | 登录抽屉 UI 按视频容器设计,含控制条 |
| 实时性 SLA | 消息 ≤ 10s、发送 ≤ 5s、状态同步 ≤ 10s;超 2× 告警 | UI 文案、超时判定、顶部告警触发逻辑 |

### V1 临时决策(未来可能微调)

| 决策点 | V1 决定 | 未来可能的变更方向 |
| --- | --- | --- |
| 权限粒度 | 按**企微号级**最细粒度,由 permission 领域承担 | 若要按"游戏/团队"分组授权,在 permission 领域内扩展,本领域接口不变 |
| 客服账号体系 | permission 领域**自维护**(账号、角色、授权关系全在内) | 后续若接外部 SSO,由 permission 领域内兼容,不影响 chat-workbench |
| 已结束会话重新联系 | 深链进入时只读；客服显式点击“重新联系”后输入区解锁，但状态机 / 指派人**不在点击时变更**，首条发送成功才跃迁到“会话中”并指派当前客服；失败或待发保持已结束 | 可结合 player-center 补“重新联系前必读字段提醒”；沉淀稳定后与 v1.1+ 智能路由一起评估统一入口 |
| `/control` 控制台 UI | 取消"控制台顶部条"和"返回工作台"按钮(顶栏 tab 已可切换)。左侧只展示已配置且已授权的企微号，不提供独立登录入口；系统管理员或运营主管在当前号投屏内、RPA 下线后可接管并操作。关闭投屏不影响左侧号卡、企微登录或 RPA | "刷新视频流/全屏/跳工作台对应会话"等在反馈明确时再补 |

### V1 明确不做

| 决策点 | V1 做法 | 去向 |
| --- | --- | --- |
| 敏感信息脱敏 | 不做,所有位置看原文 | v1.1+ 运营侧出字段清单后再加渲染层 |
| 消息撤回 | 历史方案曾跳云桌面手动撤回 | 已由顶部决定覆盖为 ChatFlow 内直接撤回请求 |

### 继续跟进的外部对齐项(不阻塞 D2)

- **permission 领域内部细节**:角色定义、企微号授权语义、登录鉴权、转接候选范围(R031),由 permission 领域 design 定义。
- **撤回服务契约**：不再依赖云桌面；待生产确认撤回时间窗、幂等和最终回执。
- **敏感信息脱敏清单**:V1 不做,运营侧给字段清单后 v1.1+ 引入渲染层。
- **会话存档 API / 无影云 API 契约**:长连消息格式、视频流协议、事件类型集合等接入细节。
- **RPA 调研原有待确认问题**:昵称 vs userid 搜索、MVP 规模、CS 资源协调等,在 [`research/v1.0-qiwei-rpa.md`](../../research/v1.0-qiwei-rpa.md) 的「待确认问题」跟踪。
