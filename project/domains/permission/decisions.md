# Permission 决策记录

> 本领域的决策/迭代历史(倒序,新在上)。`design.md` 只保留当前稳定设计;此处保留为什么这样定、范围取舍与待确认项。

## 2026-07-22 — 管理范围、最后管理员与撤回能力（覆盖同日及更早冲突口径）

- 系统管理员是平台级角色，可查看和管理全部启用游戏及相关企微号；运营主管和客服仍受游戏 + 企微号授权范围约束。服务层必须重复执行相同判断，不能只隐藏前端入口。
- 所有已认证角色都可打开 `/permission/agents` 查看预置角色说明；只有系统管理员能查看和修改客服账号与授权。未登录访问业务路由进入登录态，注销立即清理当前身份。
- 管理员可以停用或降级自己，但系统必须保证操作后仍至少有一名启用的系统管理员；只有最后一名有效管理员被保护，且服务端并发提交时再次校验。
- 普通客服仅能撤回本人已送达消息；运营主管和系统管理员可撤回授权范围内团队消息。撤回是独立业务能力，不要求通过云桌面手动完成。

## 2026-07-22 — V1 Mock 当时验收快照（后续由一致性整改覆盖）

- 需求方确认权限管理 V1 Mock 最终验收通过；三角色的功能权限、游戏与企微号数据范围、账号启停和授权配置交互不再扩展。
- 当时角色口径把撤回放在控制台、隐藏非管理员权限入口；现已由顶部决定覆盖为独立撤回能力，且三角色都可只读查看角色说明。
- 后续仅在生产对接阶段补飞书扫码身份映射、服务端强校验、持久化、并发控制与权限操作审计；这些外部依赖不反向阻塞本次 Mock 验收。

## 2026-07-22 — V1 Mock 实现快照与生产迁移项

- `/permission/agents`、本地飞书用户目录、三角色身份切换、游戏关联、企微号授权和前端守卫已实现；工作台、控制台、玩家、消息与搜索统一消费 `visibleAccountIds`。Mock 刷新后恢复种子数据，不代表真实登录态。
- 当前客服账号、飞书用户、游戏与授权保存在前端共享 Store；ops-admin 企微号配置通过兼容映射参与授权计算。阶段性实现状态统一以 `project/TASKS.md` 为准，不再写入稳定 `design.md`。
- Mock 服务写入口已补齐角色合法性、管理员数据范围、在职飞书用户与启用条件校验；指派 / 转接候选同时要求账号启用、号授权有效且当前在线，避免 UI 之外的直接调用绕过既定规则。
- 真实试运行前需补飞书扫码回调与外部身份映射、账号和授权持久化、服务端强校验、并发版本控制及权限操作审计。真实游戏目录与企微配置直接由权威服务提供，移除前端兼容镜像。
- 飞书账号候选只返回在职且尚未绑定 ChatFlow 账号的用户；提交时再次校验有效性与唯一绑定。客服账号仅展示飞书账号名，内部 `feishuUserId` 不在页面公开。

## 2026-07-22

- **控制台不再存在独立“登录企微号”入口**：企微号生命周期与资源绑定只在 ops-admin 完成。控制台左列只读取当前人有数据权限的已配置企微号；系统管理员在选中离线号、打开投屏、下线绑定机器人并进入人工接管后，才可对该号扫码登录。关闭投屏仅结束当前浏览器会话，不移除号卡，也不改变企微登录或 RPA 状态。
- **运营主管允许操作控制台**：运营主管可在已有 `visibleAccountIds` 范围投屏、接管、登录 / 退出、重启和关闭投屏；撤回已从控制台职责移出。运营主管仍不可管理账号或授权。

## 2026-07-20

- **企微号共享配置归 ops-admin，控制台只登录既有号**：ops-admin 维护企微号、所属游戏、启停和资源绑定；permission 维护访问授权。`/control` 不创建或删除配置，只承担既有号投屏与人工接管。
- **Mock 当前身份必须进入在线候选**:切换到一个启用的 Mock 身份时,同步确保其 ChatFlow `Agent` 为在线,从而可参与该号的指派或转接候选。其他种子客服在线状态仍用于模拟并发在岗;账号停用始终移出候选。
- **账号生命周期 Mock 进入共享数据源（历史口径）**:当时企微号新增、编辑、删除、游戏归属和云桌面绑定均由 ops-admin 共享 Mock Store 维护；当前已取消物理删除，改为启用 / 停用，并在状态变更后触发 permission 重算有效授权。控制台扫码只改变登录状态，不承担配置生命周期。

## 2026-07-18

- **企微号生命周期与运维操作分离（历史）**：原方案允许删除；现已覆盖为仅启用 / 停用，且撤回不属于控制台。
- **授权配置按大数据量场景收敛为搜索选择工作区**:游戏和企微号不再以 Checkbox 全量平铺。游戏使用支持 ID / 名称搜索的多选 Select;已选游戏进入固定高度左侧导航,右侧只编辑当前游戏的企微号并支持搜索、全选和清空。底部只展示游戏与企微号总数,避免长清单撑高编辑抽屉。编辑页同时改为“账号与角色 / 数据权限”两张卡片,角色使用单选下拉以降低视觉噪声。
- **客服账号页只保留业务需要的信息**:客服账号展示值直接使用飞书账号名,不复制或展示中文姓名、头像、部门等组织资料;列表移除“飞书组织”列。新建/编辑使用一个单页抽屉,飞书账号通过可搜索下拉选择,基础权限仅配置角色,账号启停统一留在列表行操作。游戏在列表、筛选、配置和预览中统一显示 `游戏ID-游戏名称`。
- **Mock 身份切换只用于前端验证,不构成登录能力**:顶栏提供系统管理员、运营主管和客服身份切换,以便演示功能权限与 `visibleAccountIds` 数据范围共同生效。它不保存真实登录态,页面刷新后恢复种子管理员;生产仍须由飞书扫码结果和服务端会话替代。
- **飞书扫码只负责身份认证,不属于 permission 的实现范围**:客服账号开通不设置初始密码,也不提供密码重置或登录页。开通时直接读取飞书用户目录并选择在职、未绑定用户;V1 用 Mock 目录和本地 Mock 当前身份模拟,后续飞书扫码返回身份后,只需映射到启用的客服账号再进入本领域的权限判定。
- **游戏是数据权限的前置隔离层**:客服账号可关联一个或多个游戏,企微号必须归属唯一游戏;只有客服与企微号先同属一个游戏,两者之间的号授权才可生效。移除客服游戏关联时,同时撤销该游戏下的有效号授权。
- **V1 游戏目录直接 Mock,不归 ops-admin 管理**:`mockGames` 和企微号 `gameId` 映射均只读服务于账号开通和权限判定,不建设游戏 CRUD 或管理页。ops-admin 仍只负责企微号与云桌面初始化。
- **控制台与云桌面是运维权限**：仅系统管理员和运营主管可访问；普通客服不能接管桌面，但可在工作台撤回本人消息。
- **当前仓库不含服务端**:V1 原型以 React/Vite 和本地 Mock 验证交互;真实身份、数据权限和外部系统调用必须由后续服务端承担,本轮不实现。
- **权限配置收敛为一个账号页和单页抽屉**:系统管理员在 `/permission/agents` 管理客服账号；新建或编辑在同一表单中配置 Mock 飞书账号、预置角色、关联游戏和游戏内企微号。游戏关联不自动授予任何号，角色也不绕过普通账号的数据范围。系统管理员自身范围现已由顶部决定覆盖为全部启用游戏和企微号。
- **功能权限与数据权限必须分层判定**:角色只回答“能否执行某个业务动作”,游戏归属与企微号授权共同回答“能操作哪些资源”;两层均通过才执行。这样调班、切换游戏或新增企微号只需调整数据关联,不需要派生新角色。
- **V1 账号体系由 ChatFlow 自维护**:客服账号、角色、游戏关联和企微号授权归 permission;飞书仅是外部身份认证来源,不替代 ChatFlow 的业务权限模型。
- **先采用预置三角色,不做权限矩阵**:系统管理员 / 运营主管 / 客服覆盖 V1 的配置、协同与一线接待;自定义角色、客服组、接待上限与客服状态机延后,避免在 MVP 阶段把权限后台做成通用 IAM。
- **游戏内企微号是最终数据范围粒度**:不引入玩家级、字段级或会话级例外授权。所有业务域先按可见游戏、再按可见企微号过滤;指派/转接候选需与会话所属号同游戏且有共同授权。
- **在线状态由外部身份会话派生**:V1 以 Mock 当前身份模拟,后续由飞书扫码会话派生;不增加手动上下线或小休状态。因此停用账号或外部会话失效都应从可指派候选移除。
- **不阻塞下一阶段的待确认项**:飞书扫码协议、回调、会话有效期、外部身份映射同步和生产审计留存进入项目级研发技术设计;V1 Mock 先使用本地种子身份与会话状态。
