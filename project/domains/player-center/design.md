# 需求业务背景

## 业务诉求

**一句话目标**:把分散在各企微号、各会话里的玩家信息,集中沉淀到一个**以玩家为主键**的档案中心,客服在接待时能快速理解玩家全貌,运营也能跨号检索玩家。

具体诉求:

1. 客服接待时能在一个面板里看到玩家的备注、标签、自定义字段、跨企微号关系、会话历史。
2. 运营能在一个独立列表页(`/players`)里检索全部玩家、批量打标签 / 改备注、跨号查询关系。
3. 玩家信息(备注、标签)可与企微原生标签 / 备注互通,同步双向不冲突。

## 概念说明

| 概念 | 定义 |
| --- | --- |
| 玩家 | 一个真实自然人;在 ChatFlow 内由唯一 `playerId` 标识。一个玩家可被多个企微号添加为好友(企微 `external_userid` 多对一)。 |
| 备注 | 客服侧填给玩家的私有名字,展示时优先于企微原昵称。可与企微原生备注双向同步。 |
| 标签 | 给玩家打的多选分类(如 "VIP-A" "高消费")。**标签库**由本领域维护,**玩家上的标签**也是本领域;企微原生标签可双向互通。 |
| 标签组 | 标签的分组(如 "等级" 组下含 VIP-A/B/C)。映射企微"客户标签组"。 |
| 自定义字段 | 运营可定义的扩展字段(文本 / 数值 / 日期 等);**字段定义**和**玩家上的字段值**都在本领域。 |
| 企微关系成员 | 一个玩家在不同企微号上的好友关系;同一玩家在 N 个号上可能各有不同的备注 / 标签。 |
| 会话记录 | 玩家维度的会话历史汇聚(跨号、跨时间);引用 chat-workbench 的会话与消息数据。 |
| 玩家档案 | 玩家在 ChatFlow 内的全部数据集合(基础信息 + 备注 + 标签 + 自定义字段 + 关系 + 会话记录)。 |

# 功能现状

## 功能当前状态

**全新领域,无历史页面。**

线下现状:运营和客服只能从企微原生客户端的"客户详情"看到玩家备注 / 标签,无法跨号汇聚一个玩家;批量打标签依赖企微后台,操作分散。

## 数据当前状态

### 数据源(真实链路)

| 类型 | 来源 | 用途 |
| --- | --- | --- |
| 玩家身份 | 企业微信"外部联系人" API | 拉取所有添加进任一企微号的好友 |
| 备注 / 标签 | 企业微信 API + ChatFlow 自维护 | 备注/标签双向同步;ChatFlow 端为权威源 |
| 自定义字段 | ChatFlow 自维护 | 企微无原生支持,完全本领域定义 |
| 会话记录 | chat-workbench 的会话 + 消息(企微会话存档 API) | 按 playerId 反向聚合 |

### V1 数据策略

- Mock 为主,真实链路留接入契约(写到 `project/tech/` 后续版本)。
- Mock 玩家规模:5-10 人,覆盖单号 / 多号关系。
- 自定义字段:V1 提供 2-3 个示范字段(如"游戏 UID""所属服"),不做完整字段管理 UI。

### 已知数据缺口

- 企微原生标签 / 备注的同步频率与冲突解决策略未定(以谁为权威源、双向同步还是单向)。
- 玩家"游戏侧"信息(UID / 区服 / 充值额) — V1 chat-workbench 已明确不做(R010-R012 留 v1.1+),本领域只放占位字段。
- SCRM 数据互通接口 — 是否需要从 SCRM 拉历史标签 / 备注?

# 功能梳理

## 功能实现思路

### 页面分层

V1 共 **2 个独立路由 + 1 个 slot**:

1. **`/players` 玩家列表页**(新增顶栏 tab"玩家")
   - 顶部:搜索 / 标签筛选 / 按号筛选
   - 主体:玩家表格(头像 / 昵称 / 备注 / 标签 / 关联号 / 最近会话时间)
   - 行操作:查看详情 / 改备注 / 打标签
   - 批量操作(V1 简化:批量打标签)
2. **`/players/:id` 玩家详情页**(独立路由,可深链)
   - 顶部:头像 + 昵称 + 备注 + 操作条(改备注 / 打标签 / 跳到对应会话)
   - 主体 Tab:**基础信息** / **企微关系** / **会话记录** / **自定义字段**
   - 同时这个页面也是右列 slot 的"完整版" — slot 渲染同一组件的精简形态
3. **chat-workbench 右列 slot**(由本领域提供 React 组件)
   - chat-workbench 传 `playerId + accountId` 上下文,本领域渲染玩家档案精简版
   - 内容来自详情页同一数据源,样式紧凑

### 关键取舍

- **playerId 是稳定主键**:跨企微号统一,真实链路用 unionid / 自有 mapping 实现;V1 用 mock 字符串。
- **标签 / 备注双源同步**:**ChatFlow 端为权威源**,企微原生改动通过 API 拉回 + 后台冲突时取最新写入。V1 不实现真同步,只 mock 双向"已互通"提示。
- **自定义字段 V1 简化**:字段定义不做管理 UI(用 mock seed 直接给 2-3 个),只做"玩家上字段值"的展示和编辑。完整字段定义 UI 放 v1.1+。
- **会话记录是只读视图**:本领域不存消息,只按 playerId 反查 chat-workbench 数据;消息内容用 chat-workbench 的渲染组件复用(避免重复)。
- **企微关系展示为主**:V1 只展示"该玩家在哪些号上、各号备注/标签"等,不做"跨号合并"或"主号选择"等复杂操作。

### 核心流程

- **列表页搜索**:输入关键词 → 玩家表格按"昵称 / 备注"模糊匹配。
- **打标签**:行操作或批量 → 弹标签选择(标签库下拉 + 当前已选)→ 写入 → 同步企微 API(V1 mock)。
- **改备注**:行操作 → 内联编辑或弹窗 → 写入 → 同步企微 API。
- **进详情**:点行 → 跳 `/players/:id`,Tab 默认"基础信息"。
- **看会话历史**:详情页 Tab"会话记录" → 列表 → 点行跳工作台对应会话(`/workbench?conversationId=...`)。
- **工作台 slot**:chat-workbench 选中会话 → 右列加载本领域组件,默认 Tab 为"基础信息";"完整档案"按钮 → 跳 `/players/:id`。

## 跨领域接口约定(初稿)

- **接 chat-workbench**:本领域提供 `<PlayerProfilePanel playerId accountId variant='slot' />` 组件,挂载在工作台右列。"完整档案"按钮路由跳 `/players/:id`。
- **接 chat-workbench(反向)**:本领域查会话记录时,从 chat-workbench 的 mock store(后续真实链路是会话存档 API)按 `playerId` 拉取;不重复存消息。
- **接 ops-admin**:不直接耦合;运营在玩家列表批量打标签,标签库由本领域自管。
- **接 permission**:client 端要拿"当前客服可见玩家集合";V1 简化为"我可见的企微号关联的玩家",由 permission 提供企微号授权列表后衍生。
- **接企业微信 API**(真实链路):备注 / 标签 / 标签组 双向同步;具体 API 与冲突策略由 `project/tech/` 的版本设计定义。
