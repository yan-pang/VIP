## P-201 玩家备注 + 玩家描述

**状态**: 新增@v1.0
**模块**: 玩家档案数据模型
**优先级**: P0
**来源**: R015 / 追加

### 业务规则

- 备注 / 描述均为**企微原生字段、绑定在(玩家×企微号)关系上**,跨号不互通;**分别展示与编辑,不合并**。
- 备注:单行短文,展示时优先于微信昵称;描述:可多行长文。
- 双向同步以 ChatFlow 端为权威源;V1 仅 mock"已同步"标识(真同步留 `project/tech/`)。
- 编辑入口仅两处:`/players/:id` 跨号关系 tab(任一管家)+ 工作台 slot(当前接待管家);`/players` 列表不暴露编辑入口、且不在主表展示备注 / 描述列(关系级字段去详情页看)。

### 页面与交互

- 详情页跨号关系 tab 单元格内联编辑(失焦 / Enter 提交,Esc 取消)→ toast"已同步至企微"。
- slot 当前管家区:备注 Input / 描述 Textarea。

### 边界与异常

- 编辑冲突(slot 与详情页同时改同一字段)→ 后写覆盖;两端各收"已被其他端更新,已应用最新值"toast,本地编辑态 patch 到最新值。

### 验收标准

- 在任一编辑入口改备注 / 描述后写入 mock store 并 toast;`/players` 列表行 / 详情页 / slot 三处实时一致。

## P-202 自定义信息

**状态**: 新增@v1.0
**模块**: 玩家档案数据模型
**优先级**: P1
**来源**: R013

### 业务规则

- 自定义信息 = 玩家维度运营自由备注,**单一文本字段** `customNote`(≤200 字),**跨企微号统一**(归属 `playerId`,不挂关系)。
- V1 取消"运营定义字段名 / 字段值"双层抽象(2026-05-30 简化);v1.1+ 接游戏数据时再考虑结构化。
- 编辑入口:`/players/:id` 基础 tab + 工作台 slot;`/players` / `/messages` 列表不暴露。

### 页面与交互

- `Input.TextArea`(autoSize 2~6 行,showCount);点击进入编辑 → 失焦保存(无独立保存按钮)→ toast"已保存"。

### 边界与异常

- `customNote` 缺失 / 损坏(mock seed 异常)→ 降级为空字符串 + 灰色 placeholder"数据异常",不阻塞其他编辑。
- 超 200 字 → 内联长度校验拦截。

### 验收标准

- 在基础 tab 或 slot 编辑 `customNote` 后,另一处实时显示同一值;长度校验生效。

## P-203 玩家标签展示与设置

**状态**: 新增@v1.0
**模块**: 标签体系
**优先级**: P0
**来源**: R014

### 业务规则

- 标签为多选分类,**绑定(玩家×企微号)关系级**;标签库由本领域自维护,可与企微原生标签双向互通(V1 mock)。
- 编辑入口:详情页跨号关系 tab + 工作台 slot;列表页只读展示(去重 chip)。
- 已废弃标签:虚态(灰底斜体)展示,不阻塞筛选(仍按原 ID 命中)。

## P-205 企微关系记录展示

**状态**: 新增@v1.0
**模块**: 关系与状态
**优先级**: P0
**来源**: R017

### 业务规则

- 底层数据形态 = (玩家×企微号)关系记录,承载该玩家在某企微号上的备注 / 描述 / 标签 / 添加时间 / 删除时间 / 关系状态。
- **`/players` 列表以玩家维度聚合**(同一玩家被 N 个号添加 → 主表只出 1 行,跨号字段去重 chip 聚合);关系级完整字段在 `/players/:id` 跨号关系 tab 按企微号一行展示。
- 每条(玩家×企微号)关系**最多对应 1 个 `conversationId`**(继承 chat-workbench"会话 ID 跨轮次不变"),故详情页把关系字段编辑与会话索引合并到同一行。

### 页面与交互

- 详情页跨号关系 tab:每行 = 一个企微号,列含 管家 / 备注 / 描述 / 标签 / 关系状态 / 添加+删除时间 / 会话索引 / 操作。详见 design.md 2.5.4。

- 单玩家仅 1 个关系 → 仍按列表渲染单行,不切卡片样式。

### 验收标准

- 列表主表玩家维度去重正确,跨号 chip 聚合无误;详情页跨号关系 tab 逐号展示完整关系字段。

## P-206 关系状态分类切换

**状态**: 新增@v1.0
**模块**: 关系与状态
**优先级**: P0
**来源**: 追加

### 业务规则

- 关系状态枚举:`normal`(正常)/ `removed_by_agent`(被管家删除)/ `removed_by_player`(被玩家删除);被删除关系保留历史数据,可只读浏览。
- `/players` 顶部三态分类切换,**默认 `normal`**;过滤语义 = "该玩家**至少有一条**该状态关系";切换保留其他筛选 + 排序,重置 page 到 1;**主表不因分类切换追加"删除时间"列**(关系级字段去详情页看)。
- 状态对主动发起的影响:`removed_by_player` → 工作台 slot 顶部红色警示横幅 + 禁用"主动发起会话";`removed_by_agent` → 橙色提示"已被管家删除,继续接待请确认意图",不禁用主动发起。
- 状态由企微 API 驱动,V1 mock seed 直接给三态样本,前端不提供"删除关系"按钮。

### 页面与交互

- 单行 chip 三态单选,每个 chip 带括号计数(与当前其他筛选联动);slot 顶部按状态显示警示 / 提示横幅。

### 边界与异常

- 玩家所有关系同状态 → 切到其他分类时该玩家不出现。
- 玩家关系跨多状态 → 切到不同分类都会出现(只要任一命中);列表不暴露具体哪条命中。

### 验收标准

- 切换分类后主表只显示"至少有一条该状态关系"的玩家,计数与列表一致;slot 对三态分别给出正确警示与主动发起按钮可用性。

## P-207 会话只读 Drawer

**状态**: 新增@v1.0
**模块**: 会话视图
**优先级**: P0
**来源**: 追加

### 业务规则

- 三处宿主(`/players/:id` 跨号关系与会话 tab、`/messages` 行操作、`/messages` `?conversationId=` 深链)**共用同一组件实例**;从右滑入 720~880px,**只读、不渲染输入区 / 发送 / 撤回**。
- 入口上下文二选一:`{conversationId}`(直接定位)或 `{playerId, accountId}`(反查最近会话,无则空态)。
- 支持**按轮过滤**:`/messages` 点某轮 → 只显示该轮(`roundIndex`);玩家详情页 / 外部深链 → 整会话(所有轮 + 分隔条)。
- 二级筛选:消息时间(范围)/ 消息内容(模糊);系统消息(轮次分隔)不被筛选过滤。发送方筛选已移除(2026-07-18,避免与列表级"最后消息发送方"重复且 Drawer 内单会话意义不大)。
- 底部"去工作台接待 →"按宿主上下文走两条 query:已绑定 `conversationId` → `/workbench?conversationId=`;空态新会话 → `/workbench?playerId=&accountId=`;落库语义由 chat-workbench 承担。

### 页面与交互

- 头部:玩家昵称 + 备注 / 企微号徽标 / 会话标识(`conversationId` 或 `conversationId#N`,等宽 + 复制)/ 关闭;主体复用 chat-workbench 消息气泡组件(继承气泡 8px / 行高 1.55 例外)。详见 design.md 模块 4。
- **消息记录展示口径**(继承 chat-workbench P-122):气泡显示实际回复客服姓名 + `HH:MM`(悬停在时间上原地展开为完整 `YYYY-MM-DD HH:MM:SS`,不用 Tooltip);按自然日插入日期分割线;企微号为会话级固定值,展示于头部徽标而非逐条气泡。

### 边界与异常

- 空态(从未产生会话)→ `Empty`"该玩家与该企微号暂无会话记录" + "去工作台主动发起"按钮;头部不显示 conversationId。
- 错误态(`conversationId` 不存在 / 超权限)→ `Empty`"会话不可见或不存在…" + 底部按钮隐藏。
- 失败气泡 ❗ hover 显示原因但**不响应点击**(只读语义,不打开失败抽屉)。

### 验收标准

- 三处宿主打开的是同一组件、交互一致;按轮过滤与整会话两种模式正确;"去工作台接待"按上下文走对 query。

## P-208 会话记录反查

**状态**: 新增@v1.0
**模块**: 会话视图
**优先级**: P0
**来源**: R016

### 业务规则

- 本领域**不重存消息**,只按 `playerId` / `conversationId` 反查 chat-workbench 会话存档。
- **会话反查源 = 工作台同款集合**:chat-workbench 静态 seed **+ 已落库的主动发起会话**(共享持久化模块 `proactiveStore`,落库判定 = 非 seed 会话有 ≥1 条 `sent` 的 outgoing 消息);会话索引 / 轮次 / 消息反查**懒计算合并两者**。
- 确保 `/players/:id` 总会话数、跨号关系 tab"查看会话 / 主动发起"、`/messages` 列表、会话只读 Drawer 与工作台左列**会话集合完全一致**(此前只读 seed 会漏掉已落库的主动发起会话)。

### 页面与交互

- 详情页跨号关系 tab"会话"列、`/messages` 表格、Drawer 消息体均来自该反查。

### 边界与异常

- 反查失败 → 宿主局部错误态 + 重试,不清空已加载数据。

### 验收标准

- 同一会话在工作台左列与本领域三处反查视图中一致出现(含已落库的主动发起会话,如演示账号"大海")。

## P-209 玩家管理列表页(`/players`)

**状态**: 新增@v1.0
**模块**: 页面承载
**优先级**: P0
**来源**: 追加

### 业务规则

- 顶栏 tab"玩家管理";**玩家维度表格**(以 `playerId` 去重,一行一个真人),**纯只读、不展开、点行不跳转**(仅行高亮)。
- 顶部筛选条 6 项(playerId 精确 / 微信昵称模糊 / 管家多选 / 玩家备注模糊 / 玩家标签多选 / 玩家描述模糊),**每条筛选命中"该玩家任一可见关系"即纳入**(玩家级 OR 语义),debounce 300ms 即时刷新,无需"搜索"按钮。
- 关系状态分类切换见 P-206;行操作仅一个"查看完整档案"(同 tab 跳 `/players/:id`,深链可分享);**不再触发会话只读 Drawer**。
- 筛选 / 分类 / 排序 / 分页态实时回写 URL(`replaceState`,默认值省略),支持深链 / 刷新 / 分享还原。
- 可排序字段白名单 = `latestUpdatedAt`(默认 desc)/ `earliestAddedAt` / `relationCount`;排序受控(跨页排序正确)。

### 页面与交互

- 列(1440 基线):微信昵称(头像+昵称+playerId 副标题)240 / 添加管家 chip 220 / 玩家标签 chip 220 / 关系数 90 / 最早添加 110 / 最近更新 130 / 操作 130。详见 design.md 页面 1。
- 分页默认 20,可切 10/20/50/100;计数文案"共 N 位玩家"。

### 边界与异常

- 筛选无结果 → `Empty`"未找到匹配的玩家" + "清空筛选"(只清筛选不切分类)。
- 完全无可见关系 → `Empty`"暂无可见的玩家,联系管理员授权企微号"。
- "清空筛选"不重置分类 chip 与排序。

### 验收标准

- 玩家维度去重、跨号 chip 聚合、玩家级 OR 筛选、关系状态分类、受控排序、URL 深链还原均正确;列表无编辑入口、不触发 Drawer。

## P-210 玩家详情页(`/players/:id`)

**状态**: 新增@v1.0
**模块**: 页面承载
**优先级**: P0
**来源**: 追加

### 业务规则

- 真人级容器,以 `playerId` 为主键、跨号聚合;**为 v1.1+ 游戏数据预留 tab 容器**。
- V1 双激活 tab + 1 占位 tab:**基础**(`basic`,自定义信息可编辑 + 备注/描述聚合只读视图)/ **跨号关系与会话**(`relations`,关系字段内联编辑 + 会话索引 + 行操作)/ **游戏数据**(`game`,v1.1+ 占位,可点击不灰显,**不列待接入字段清单**)。
- 关系级字段编辑**主入口**在 `relations` tab(每条关系最多 1 个 conversationId,合并同行);原"会话记录"tab 已并入 `relations`,深链 `?tab=sessions` 兼容重定向到 `relations`。
- 页头跨号汇总:主徽章按 `N_visible` 展示,`M_invisible>0` 时附灰色"+M 不可见";总会话数仅统计可见关系。
- Tab 切换 `replaceState` 回写 `?tab=`,不入栈;切换不重拉(本地缓存)。

### 页面与交互

- 页头(头像 64 / 昵称 / playerId 复制 / 三汇总徽章)+ 返回按钮 + 面包屑;主内容区 1440 基线限宽 1280 居中。
- `relations` tab 行操作三态:有会话 → "查看会话"(开 Drawer);正常无会话 → "主动发起"(跳工作台 `?playerId&accountId`);被玩家删除 → "主动发起"禁用 + tooltip。详见 design.md 页面 2。

### 边界与异常

- playerId 不存在 → 错误态"玩家不存在或已被清理" + 返回玩家管理按钮。
- `N_visible=0` 且 `M_invisible>0` → 汇总徽章 0 + "+M 不可见";`relations` tab 空态提示联系管理员授权。
- `?tab=` 不合法 → 静默落默认 `basic`。

### 验收标准

## P-211 消息管理列表页(`/messages`)

**状态**: 新增@v1.0
**模块**: 页面承载
**优先级**: P0
**来源**: 追加

### 业务规则

- 顶栏 tab"消息管理";**维度 = 会话轮次**(`roundId = conversationId#N`,同一会话多轮拆多行,单轮也带 `#1`;按 system「本次会话已结束」切分,纯展示派生非持久化)。
- 顶部筛选条:会话标识(精确,支持 conversationId 命中全部轮次 / roundId 命中单轮)/ 对话管家多选 / 消息时间范围 / 消息内容(模糊,**按该轮**扫 text+mediaName,debounce 500ms)/ 玩家备注模糊 / 玩家标签多选。(2026-07-18 移除"最后消息发送方"筛选)
- 行操作"查看对话消息"→ 开会话只读 Drawer **按轮过滤**(只显示该轮);**"去工作台接待"按整会话 `conversationId` 跳**(URL 用 `?conversationId=<cid>&round=<n>` 两参数,不把含 `#` 的 roundId 塞进 query)。
- `?conversationId=` 深链:进入即**异步并行**打开 Drawer(整会话)+ 表格按其余筛选渲染,二者互不阻塞。
- 筛选 / 排序 / 分页回写 URL(与 Drawer 的 `conversationId`/`round` 参数共存,筛选会话标识用 `cid` 区分);可排序白名单 = `lastMessage.sentAt`(默认 desc)/ `messageCount`,受控排序。

### 页面与交互

- 列(1440 基线):会话轮次 200 / 玩家 235 / 对话管家 140 / 玩家标签 155 / 最后消息 420 / 消息总数 85 / 操作 165。详见 design.md 页面 3。
- **消息记录展示口径**(继承 chat-workbench P-122):行内三要素齐备 —— **对话管家列 = 回复的企微号名称**;**最后消息 = 实际发送方**(agent 显示实际回复客服姓名、player 显示"玩家")**+ 完整日期时间 `YYYY-MM-DD HH:MM`** + 内容预览。

### 边界与异常

- 筛选无结果 → `Empty`"未找到匹配的会话" + 清空筛选(不重置排序、不关 Drawer)。
- 深链 `conversationId` 不存在 / 超权限 → Drawer 错误态,表格正常渲染。
- 消息内容全文索引超时 → 表格降级为"按其余筛选命中" + 顶部告警。

### 验收标准

- 轮次维度拆分正确;会话标识筛选支持 conversationId / roundId;按轮 Drawer 与整会话深链行为正确;"去工作台接待"按整会话跳;URL 与 Drawer 参数共存还原。

## P-212 工作台 slot 精简档案

**状态**: 新增@v1.0
**模块**: 页面承载
**优先级**: P0
**来源**: 追加

### 业务规则

- 本领域提供独立组件 `<PlayerSlotPanel playerId accountId />`,挂 chat-workbench 右列;**不复用 `/players/:id`**,内容更少更聚焦本次接待。
- 字段范围(只装强相关):头像 / 昵称 / **当前管家**备注+描述+标签(可编辑)/ 关系状态徽标 / 跨号 chip(只显示其他企微号名、不展开)/ 自定义信息(单一 `customNote`,与详情页基础 tab 同源)。

- popup blocker 阻挡新标签页 → toast"请允许在新标签页打开"。

### 验收标准

## P-213 跨页数据同源与实时同步

**状态**: 新增@v1.0
**模块**: 数据同源
**优先级**: P0
**来源**: 追加(贯穿 P-201/P-202/P-203/P-205/P-206)

### 业务规则

- 三处视图(`/players` 列表行 / `/players/:id` 跨号关系 tab + 基础 tab 聚合视图 / 工作台 slot)**共用同一份玩家与关系 mock store**。
- 任一处编辑(备注 / 描述 / 标签 / 关系状态 / 自定义信息)→ 写入 store → **广播变更** → 所有视图实时刷新,无需手动刷新页面。
- 真实链路:企微原生备注 / 描述 / 标签 / 标签组双向同步,ChatFlow 端为权威源;V1 仅 toast"已同步至企微"(关系字段)/"已保存"(自定义信息)。
- 编辑冲突(多端同改同字段)→ 后写覆盖 + 双方"已被其他端更新"toast + 本地编辑态 patch。

### 页面与交互

- 通过 `subscribePlayerCenter` 广播机制驱动各视图重渲染(列表聚合 chip、详情页行、slot)。

### 边界与异常

- 广播未到达前 slot 显示自己写入的最新值;到达后若冲突 → patch + toast。
- mock 写入失败 → toast + 回滚到原值,不污染其他视图。

### 验收标准

- 在 slot 改备注 → `/players` 列表对应会话 / 行显示名与详情页同字段在无刷新下即时变化;清空备注回落昵称;三处终态一致。
